## 目的

当社（株式会社mijica、フリーランス特化のITエージェント／SES事業）は、常に **クライアント（発注元）** と **稼働者（フリーランス／BP人材）** の間に挟まれている。
片方から相談・要望・回答が来たら、もう片方に相談・確認し、その結果をまた片方に返す——というやり取りが頻繁に発生する。

このスキルは、**双方のインプット（2ソース、または片側）を読み込み、文脈・時系列・各者の要望・争点を把握したうえで、適切な相手向けの返信案を作成する**。
最大の肝は「**片方の生文面をそのまま相手に転送しない**」「**実態と乖離する主張をして反論余地を作らない**」こと。

## ワークフロー進捗

このチェックリストをコピーして進捗を追跡する:

```
- [ ] ステップ1: インプット受け取り（クライアント側／稼働者側）
- [ ] ステップ2: 各ソースのテキスト化（Slack / Gmail / テキスト）
- [ ] ステップ3: 文脈・時系列の把握（商流・争点・各者の要望・現フェーズ）
- [ ] ステップ4: 仮説立案 → どちら向けの返信か＋各送付先の媒体・テイストを提案して確認
- [ ] ステップ5: 返信案の作成（相手別の作成原則・媒体・テイストに従う）
- [ ] ステップ6: セルフチェック（文面単体＋過去の流れとの論理整合／既定軽め・要望時に深掘り）
- [ ] ステップ7: 出力（媒体に合わせた形式＋pbcopy、タスク配下なら output 保存）
```

## 前提

- 実行環境: Claude Code 上の手動実行。判断ロジックは Claude 本体が担う。
- カレントディレクトリがタスクフォルダ配下なら `output/` に成果物を保存する。そうでなければ画面提示＋pbcopy のみ。
- グローバル設定（`~/.claude/CLAUDE.md`）の機微情報・外部通信ルールを遵守する。**送信は行わない。文案を作って本人に渡すまで**がスキルの責務。

## 入力ソース

| # | ソース | 渡し方 | 必須/任意 |
|---|--------|--------|---------|
| 1 | クライアント側の文脈 | Slack URL / Gmail URL / テキスト直貼り / ファイルパス | いずれか1ソース以上は**必須** |
| 2 | 稼働者側の文脈 | Slack URL / Gmail URL / テキスト直貼り / ファイルパス | 文脈に応じて（双方読むのが基本） |
| 3 | 補足（注文書・契約・過去ログ等） | テキスト / ファイルパス / URL | 任意 |

- インプットが片側しかなくても、もう片側の過去ログ（Slack/Gmail）を辿れる場合は辿って文脈を補う。
- 何も渡されず曖昧な依頼の場合は、「どちらとどちらのやり取りか、Slack URL かテキストを貼ってください」と促す。

### ソースの読み取り方

- **Slack URL**（`https://<workspace>.slack.com/archives/<CHANNEL_ID>/p<digits>`）:
  - `CHANNEL_ID` を抽出（`C…`=チャンネル、`D…`=DM、`G…`/mpim=グループ）。
  - `p<digits>` → 末尾6桁の前にドットを入れて `message_ts` に変換（例: `p1781611568561169` → `1781611568.561169`）。
  - スレッドは `mcp__claude_ai_Slack__slack_read_thread(channel_id, message_ts)`、前後の文脈は `mcp__claude_ai_Slack__slack_read_channel(channel_id)`。DM は `channel_id` に DM チャンネル ID（`D…`）または相手の user_id を渡す。
  - これらは遅延ツール。未ロードなら `ToolSearch` で `select:mcp__claude_ai_Slack__slack_read_thread,mcp__claude_ai_Slack__slack_read_channel` を先に読み込む。
- **Gmail URL**（`https://mail.google.com/mail/u/0/#…/<id>`）:
  - 末尾の id を `thread_id` として `mcp__claude_ai_Gmail__get_thread` で取得（未ロードなら `ToolSearch` で `select:mcp__claude_ai_Gmail__get_thread`）。
  - 添付が必要な場合は MCP では取得不可。`download_gmail_attachments(thread_id, out_dir)`（`~/.claude/snippets/google_workspace.py`）を Python で実行する。
  - **WebFetch / gws CLI は使わない**（認証が通らない）。
- **テキスト直貼り / ファイルパス**: そのまま読む。

## ステップ3: 文脈・時系列の把握

返信案を書く前に、必ず次を言語化する（簡潔に内部整理）:

- **商流・登場人物**: クライアント（発注元）／当社（mijica）／稼働者。窓口担当者名も拾う。
- **時系列**: いつ・誰が・誰に・何を言ったか。特に「直近どちらの番（ボール）か」。
- **争点**: 何が論点か（条件・契約・運用・トラブル・金額・スケジュール等）。
- **各者の要望・温度感**: それぞれが何を求め、どの程度強く言っているか（更新停止検討・不満・歩み寄り等）。
- **すでに合意できた点／未決の点**。

## ステップ4: 仮説立案 → 確認（媒体・テイストも最初に確認）

時系列・文脈から仮説を立て、**作成に入る前にまとめて確認する**。確認するのは次の3点:

1. **どちら向けの返信か**: 「直近どちらの番か」から仮説を立て提示する。片側が自然ならその旨を述べて確認、両面が必要そうなら両面を提案。
2. **各送付先の媒体**: 返信案を作る各相手について、送付媒体（**メール / Slack / LINE / Chatwork / その他テキスト**）を確認する。
   - インプットのソースから媒体が明らか（例: Slack スレッドの続き → Slack）な場合は、その推定を述べたうえで確認する。
   - **判別できない場合は必ず聞く**（媒体で文面の形式が変わるため。メール＝件名・宛名・フォーマル／Slack・LINE＝口語・中黒箇条書き）。
3. **各送付先のテイスト**: 相手ごとにどのトーンで行くかを確認する（例: フォーマル堅め／丁寧だが柔らかめ／フランク寄り／強めに要望、など）。過去ログがあれば既存トーンを推定して提示し、それでよいか確認する。

> 例: 「直近クライアントから回答が来ているので、稼働者向けに整理した返信を作るのが自然と見ています。
> ・宛先: 稼働者1通でよいですか？（両面も可）
> ・媒体: これまで Slack DM のやり取りなので Slack 想定でよいですか？
> ・テイスト: いつもの"丁寧だが柔らかめ・関係維持優先"で行きますか？」

- ユーザーの指示が明確（宛先・媒体・テイストが既に分かっている）なら、確認は簡略化してよい。

## ステップ5: 返信案の作成（相手別の原則）

### 共通原則（最重要）

1. **生文面の転送禁止**: 一方（特にクライアント）の回答・文面を**そのまま相手に貼らない**。当社の整理・言葉に置き換える。
   - 例: クライアントの「業務遂行状況・成果の報告に代わるもの」のような表現は、稼働者にそのまま渡すと反発を招く。当社の整理に翻訳する。
2. **実態と乖離する主張をしない（反論余地を作らない）**: 「全部当社が間に入る」など、現実と矛盾する約束・説明をしない。事実と整合する範囲で書く。
3. **まず実利を前面に**: 結論・合意できた点・条件（朗報）を先に。機微・法的・込み入った論点は**突っ込まれない限り深掘りしない**（軽く触れる程度）。
4. **各者の要望を取りこぼさない**: 相手が出した質問・要望に漏れなく応答する。
5. **確認した媒体・テイストに合わせる**: ステップ4で確認した媒体（メール／Slack／LINE 等）の形式と、相手ごとのテイストで書く。過去ログがあれば文体・距離感も踏襲する。
6. **過去に送った文面と矛盾しない**: これまでの自分（当社）の発言・約束・整理と論理的に一貫させる。前言と異なる立場や、過去の説明と噛み合わない主張を新たに出さない。
7. **日本語として自然か（敬語・冗長表現）**: ビジネス文として違和感のない自然な日本語にする。送る前に下記を必ず点検する。

### 日本語の自然さ（敬語の付けすぎ・冗長表現を避ける）

機械的に丁寧表現を盛ると、かえって不自然になる。以下に注意する:

- **「お／ご」の付けすぎ**: 美化語が馴染まない語に「お／ご」を付けない。
  - ✕ お役割 → ◯ 役割／ご担当範囲
  - 目安: 「お役割」「お案件」のように、日常で美化語が定着していない語へ「お」を付けない。「ご担当」「ご希望」など定着している語はOK。迷ったら付けない方が自然。
- **冗長・曖昧な名詞の連結**: 意味が重なる語や曖昧な語を「・」で並べない。一語で言い切る。
  - ✕ 連携・サポートの進め方 → ◯ サポート体制／連携の進め方（どちらか一方に絞る）
  - ✕ 確認・すり合わせをさせていただく → ◯ すり合わせさせていただく
- **二重敬語・回りくどい言い回し**: 「させていただく」の多用、「〜のような形で」等の曖昧表現を削る。意味が同じなら短い方を選ぶ。
- 迷ったら、**自分が口頭で言って違和感がないか**を基準にする。

### クライアント向け

- **丁寧・フォーマル**（です/ます、ビジネス文書寄り）。
- 稼働者の懸念・要望を**整理して客観的に**伝える。稼働者の生の強い表現（「ストレス」「更新やめる」等）はそのまま出さず、「ご負担」「今後の参画継続にも影響し得る」等にトーンダウンして影響・リスクとして提示する。
- 依頼・相談は具体的に（何を、どこまで、いつまでに調整したいか）。

### 稼働者向け

- **丁寧だが柔らかめ・関係維持優先**（Slack DM ならなおさら口語寄り）。
- **朗報（緩和・合意できた点）を主軸**に。
- 懸念には「おっしゃる点は理解しています／一理あります」と受け止めつつ、**深追いしない**。
- クライアントの内部事情・生回答はそのまま渡さず、当社の整理として伝える。

## ステップ6: セルフチェック（既定: 軽め）

返信案ができたら、**(A) 文面単体** と **(B) これまで送った文面の流れ** の両面で点検する。

**(A) 文面単体の点検**
- [ ] **実態と矛盾**する主張はないか（現実にやっていること／いないことと整合するか）。
- [ ] 相手から**即返せる反論余地**はないか（特に契約・商流・指揮命令などの論点）。
- [ ] 一方の**生文面・機微情報**を不用意に相手へ持ち出していないか。
- [ ] 各者の**要望・質問に漏れなく**応答しているか。
- [ ] トーン・媒体形式は相手と関係性に合っているか。
- [ ] **日本語として自然か**（不自然な美化語＝「お役割」等／冗長な名詞連結＝「連携・サポートの進め方」等／二重敬語がないか）。口頭で言って違和感がないかで判断する。

**(B) これまでの流れとの論理整合（毎回必須）**
これまで当社が双方に送ってきた文面の流れを振り返り、**ロジックに穴がないか**を点検する:
- [ ] 過去に当社が示した立場・整理・約束と、今回の文面が**矛盾していない**か（前言撤回・ダブルスタンダードになっていないか）。
- [ ] クライアント向けに言ったことと稼働者向けに言ったことが**食い違っていない**か（両面で別々の説明をして後で破綻しないか）。
- [ ] 今回の主張が、過去のやり取りで相手から既に指摘された論点を**蒸し返して**いないか／相手に**新たな攻撃材料**を与えないか。
- [ ] 一連の流れとして話が**前進している**か（堂々巡り・論点のすり替えになっていないか）。
- [ ] 過去の経緯・事実関係（日付・合意内容・誰が何を言ったか）と**齟齬がない**か。

→ 穴が見つかったら、それを塞ぐ形に文面を直す。流れの中で説明が苦しくなる場合は、無理に押し切らず「ここは整理し直す必要がある」とユーザーに指摘する。

**チェックで問題が見つかったら、出力する前に文案を修正し、再度この観点で点検する**（検証→修正→再検証）。問題がなくなってからステップ7へ進む。

**法的・機微な論点（偽装請負・契約解釈・労働者性・金額/解約トラブル等）が絡む場合は、深追いせず軽く触れたうえで、ユーザーに「ここは深掘りしますか？」と提案する。**
ユーザーが希望した場合のみ、必要に応じて法令・一次情報の確認（リサーチはサブエージェントに委譲）を行い、反論余地を精査して整理を補強する。

> 参考: 偽装請負・労働者性が論点になった場合の着眼点
> - フリーランスが**一人親方（従業員なし）**なら classic な偽装請負（労働者派遣の偽装）は定義上ほぼ成立しない。実質論点は①**労働者性**②**報告先（民法645条＝委任者）**。
> - 「実績の記録・報告」は「業務の指示（指揮命令）」とは別物。判断軸は「発注者が稼働時間・進め方を社員のように管理・是正しているか」。
> - 一次情報: 37号告示（昭和61年労働省告示37号）、疑義応答集、昭和60年労基法研究会報告。

## ステップ7: 出力

- 返信案を**コードブロック**で提示し、そのまま貼れる形にする。**確認した媒体に合わせて形式を変える**:
  - **メール**: 件名案 → 宛名（◯◯様）→ 本文 → 結びの順。フォーマル。署名は本人が付けるため入れない（必要なら `[署名]` プレースホルダ）。
  - **Slack / LINE / Chatwork 等のテキスト**: 件名・宛名は不要。冒頭は相手名の呼びかけ程度。箇条書きは**中黒（・）**。口語寄りで、絵文字は既存トーンに合わせて控えめに。
- 両面作る場合は「【クライアント向け｜媒体】」「【稼働者向け｜媒体】」と見出しを付けて2つ提示する。
- 最後に **pbcopy** で確認する。複数案・両面あるときは「どれをコピーしますか？」と聞く。

```bash
cat <<'EOF' | pbcopy
（返信案本文）
EOF
echo "copied"
```

- カレントがタスクフォルダ配下なら、`output/` に Markdown 保存する（命名規則: `{ディレクトリ名}_{内容}.md`）。

## やってはいけないこと

- クライアント／稼働者の生文面・機微情報をそのまま相手に転送する。
- 現実（実際の運用・やり取り）と矛盾する説明・約束をする。
- 機微・法的論点を、頼まれていないのに長々と説明して文案を重くする。
- ユーザーの確認なしにメール・Slack を送信する（**作成までが責務**）。
- 一方の私的・健康・金銭等の機微な発言を他方に持ち出す。
