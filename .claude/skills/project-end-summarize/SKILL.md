## 目的

当社（株式会社mijica、フリーランス特化のITエージェント／SES事業）が、**エンド企業から直接来た案件**（自社の直案件・エンド直）をフリーランスへ提案する際に使う。
ユーザーがターミナルに貼った**未整形のエンド案件情報（テキスト or URL）**を、**企業名・サービス名をマスキング（一般化）**したうえで、提案用の**指定フォーマット（FMT）**へ正確に整形して返す。

- `project-bp-summarize`（BP＝パートナー企業紹介案件）とは**別スキル**。エンド直案件は企業名・サービス名・プロダクト名が生のまま含まれるため、**マスキングが本スキルの核**。
- BP版とは整形ルールに差分がある（不明項目・外国籍・商流・案件名ラベル等）。本スキルのルールを優先する。

## 入力

1. ターミナルに貼られた未整形のエンド案件情報（自由形式テキスト）。
2. **案件URL**（HERP 等の ATS・媒体）。**Claude が WebFetch で本文を取得**して整形する。
   - HERP Careers は公開ページ（ログイン不要）。求人詳細の JSON-LD（`script[type="application/ld+json"]`）に `hiringOrganization.name`（企業名）と `description`（案件内容・スキルを含むHTML）がある。HERP はあくまで一例で、他の ATS・媒体の可能性もある。
   - WebFetch で取得できない場合（ログイン要・JSステート依存等）は、その URL について「本文を貼ってください」と促す。**取得できないのに推測で埋めない**。
3. **複数案件（複数URL／複数本文）が渡されたら、案件ごとに個別に整形**し、コードブロックを案件ごとに分けて出力する。
4. 案件情報が無く曖昧な依頼なら「整形したいエンド案件情報（テキストまたはURL）を貼ってください」と促す。

## マスキングルール（本スキルの核・最優先）

エンド直案件は社名・サービス名がそのまま含まれる。提案文面に出る前に必ず一般化する。

### 案件名
次の情報をもとに `【メイン技術/リモート頻度/週何日の稼働か】業務内容を表す一般名` の形式で生成する。

- **メイン技術**: 主に開発に使う技術/FW を1〜2個（例: React、Laravel、Python）。2つはスラッシュで併記（例: `Python/Next.js`）。
- **リモート頻度**: `フルリモート` / `リモート併用` / `常駐` のいずれか。
- **稼働日数**: `週5日` `週3〜5日` 等、募集内容に合わせる。
- **業務内容の一般名**: 企業名・サービス名を使わず、業務内容を自然な名称にする（例: 「業務改善SaaS開発支援」「人事領域の業務プロセス開発」）。
- カッコ `【】` はそのまま残す。各項目の区切りは**半角スラッシュ `/`**（全角 `／` 禁止）。不要なスペースを入れない（`Python / Next.js` → NG、`Python/Next.js` → OK）。
- 言語とFWを併記する際は半角カッコ: `PHP(Laravel)`。
- **出力行にラベル `【案件名】` は付けない**。区切り線の間に値だけを置く。

### 案件の内容
- 製品名・プロダクト名・社名は使わず、SaaSカテゴリ／機能名で一般化する。
- 具体的なアプリ名・Webサービス名・動画配信名・ゲームタイトル等は記載不可。
- 会社/事業の Vision・Purpose にあたる文言は使用しない。
- 経費精算・入金消込などの具体名は「経理業務の各プロセス」等へ置換。
- **入力中の「弊社」「当社」や社名・サービス名は一般語へマスキング**する（「同社」「開発チーム」「事業会社」等）。エンド直案件では「弊社」がエンド企業自身を指すことが多いため、生のまま残さない。
- 300〜400字程度、やや丁寧なです・ます調、適度に改行（1文ごと／意味の区切りごと）。「〜いただきます」が繰り返されないよう文調を整える。
- スクラム等の開発ルール・開発体制の記述は、備考ではなく**「案件の内容」欄**に含める。

### 求めるスキル／歓迎スキル
- 特定の企業名・資格名は使わない。
- 例: ERP導入 → 「業務系SaaS／業務システムの導入経験」のように抽象化。
- 基本1項目ずつ展開し、圧縮・省略しない（ただし似た文言は集約してよい。例: 「バックエンド・フロントエンド3年以上」）。

### 開発環境
- 見出しは `言語：/FW：/インフラ：/その他：` の4サブ項目で記載する。
- 各値は特定技術を1〜2語に抑えつつ、括弧や分類語で補足してよい（例: `TypeScript(React/Next.js)`、`AWS(ECS/Fargate/Lambda)`、`Golang/PHP(Laravel)`）。
- 入力に開発環境の情報が**一切無ければ** `■開発環境` セクションを**見出しごと丸ごと削除**する。1つでも情報があればセクションを残し、情報の無いサブ項目（言語/FW/インフラ/その他）はその行を削除する。

## 整形ルール（マスキング以外）

1. **支払いサイト_締め日** は必ず `40日サイト (当月末締め翌々月10日支払い)` に固定（入力に別記載があっても固定）。
2. **単価はスキル見合いレンジへ変換**（下記「単価変換ロジック」を厳守）。
3. **稼働日数** は入力に特段の言及（週4日可・週3日等）が無ければ `週5日`。別条件があればその値に絞る。
4. **稼働形態** は原文のニュアンスを保持（`フルリモート`/`リモート併用`/`常駐` など書かれたまま）。
5. **コアタイム・フレックス等の勤務時間に関する記述**は、備考ではなく**「作業開始/終了時間の目安」欄**に記載する。
6. **商流行（■商流）は入れない**（エンド直案件のため）。
7. 箇条書きは改行・句読点を整えて読みやすくする。長文項目（案件の内容）は句点・文の切れ目で改行し、1行が長くなりすぎないようにする。
8. 提案資料やドキュメントファイルとして出力せず、**チャットベース（コードブロック形式）**で返す。
9. FMT を記号・見出し・固定文言を含めて正確に出力する。

## 単価変換ロジック（整形ルール2の詳細）

提示単価から **下限側は15万円、上限側は10万円** を差し引いてレンジ化し、
`スキル見合い（{下限} ~ {上限}万/税別）` の形式で記載する。

- **単一値で提示**: `X万` → 下限 = X−15、上限 = X−10
  - 例: 90万 → `スキル見合い（75 ~ 80万/税別）`
  - 例: 80万 → `スキル見合い（65 ~ 70万/税別）`
- **レンジで提示**: `下限L 〜 上限H 万` → 下限 = L−15、上限 = H−10
  - 例: 90〜100万 → `スキル見合い（75 ~ 90万/税別）`
  - 例: 70〜80万 → `スキル見合い（55 ~ 70万/税別）`
- **ベースレンジ＋「上振れ」「max◯万」「スキルセット優先」等の補足が付く場合**: まずベースレンジに上記ルールを適用して算出し、その上で「上振れ・max を上限に反映するか」を**出力後の要確認点として提示**する（勝手に上限を引き上げない）。ユーザーが提示レンジをそのまま採用する等の指示を出した場合はそれに従う。
- **税込/税別の区別はしない**。提示値をそのまま差し引きに使う（÷1.1 はしない）。`/税別` 表記は固定で常に維持する。
- **単価の記載が無い場合**は計算せず `スキル見合い（● ~ ●万/税別）` のまま残す。

## 不明項目の扱い

- **不明な項目は `●` で残す**（行ごと削除しない）。
- `■外国籍`・`■希望する年齢層` も**FMT に残す**（不明なら `●`）。
- 例外は開発環境セクションのみ（情報皆無時に丸ごと削除。上記「開発環境」参照）。

## 出力フォーマット（FMT）

以下は**完全形のスケルトン**。案件名行はラベルなし・値のみ。各 `●` を入力に応じて置換し、不明項目は `●` のまま残す（開発環境のみ情報皆無時に丸ごと削除）。出力は必ずコードブロック（```）で囲む。

```

【React/フルリモート/週5日】業務改善SaaS開発支援
-------------------------------
■案件の内容
●

■求めるスキル：
・●
・●

■歓迎スキル：
・●
・●

■開発環境：
言語：●
FW：●
インフラ：●
その他：●

■稼働日数：週5日
■支払いサイト_締め日：40日サイト (当月末締め翌々月10日支払い)
■商談：●回 (オンライン)
■募集人数：●人
■開始時期：●月
■稼働形態：フルリモート/リモート併用/常駐
■単価：スキル見合い（● ~ ●万/税別）
■精算幅：●h - ●h
■作業開始/終了時間の目安：●:00-●:00
■希望する年齢層：●~●歳
■外国籍：可,不可
■PC貸与：●
■備考：
●
-------------------------------
```

### FMT に関する補足
- 1行目の案件名は `【メイン技術/リモート頻度/稼働日数】業務内容の一般名`（上記スケルトンは例）。
- `■稼働形態` のように選択肢が併記される項目は、入力で特定できればその値だけに絞る。特定できなければ `●` で残す（または原文ニュアンスを保持）。
- `■稼働日数` の既定は週5日。

## 出力前セルフチェック

出力する前に、以下を必ず確認する:

- [ ] 企業名・サービス名・プロダクト名・Vision/Purpose 文言が残っていないか（**マスキング漏れは致命的**）
- [ ] 「弊社」「当社」「社名」を一般語へ置換したか
- [ ] 案件名が `【技術/リモート頻度/稼働日数】業務内容の一般名` 形式・半角スラッシュ・ラベルなしか
- [ ] `■支払いサイト_締め日` が「40日サイト (当月末締め翌々月10日支払い)」固定か
- [ ] 単価の差し引きが正しいか（下限 −15万 / 上限 −10万、`/税別` 表記維持）
- [ ] 開発環境が `言語/FW/インフラ/その他` 形式で、情報皆無時はセクションごと削除したか
- [ ] 不明項目を `●` で残しているか（行削除していないか）
- [ ] `■外国籍`・`■希望する年齢層` を FMT に残しているか
- [ ] 商流行（■商流）を入れていないか
- [ ] 案件の内容が300〜400字・です・ます調・改行整形済みか
- [ ] コアタイム・勤務時間が「作業開始/終了時間の目安」欄に入っているか
- [ ] FMT の区切り線・見出し記号（`■` `・`）が欠けていないか
- [ ] コードブロック（```）形式で返しているか／複数案件は案件ごとに分けたか

## 出力後

- 整形結果（コードブロック）に続けて、`●` のまま残した不明点・入力の矛盾・要確認点を簡潔にリストで提示し、「不足情報があれば追記して再整形できます」と添える。
- 単価を差し引き変換した場合は、計算根拠（例: 提示90万 → 75〜80万）を1行で補足する。
- **最後に必ず「pbcopy しますか？」と確認する**。承認されたら、整形結果本文（区切り線から区切り線まで＝コードブロックの中身、バッククォート ``` 自体は含めない）を `pbcopy` でクリップボードへコピーする。
  - コピー方法: 整形結果テキストを `pbcopy` に渡す（例: `cat <<'EOF' | pbcopy ... EOF`、または一時ファイル経由で `pbcopy < file`）。
  - **複数案件のときはどの案件をコピーするか確認**してからコピーする。
  - 再整形のたびに最新の結果で再確認する。
