## 目的

当社（株式会社mijica、フリーランス特化のITエージェント／SES事業）が、パートナー企業（BP＝ビジネスパートナー＝懇意先）から紹介されたフリーランス人材を**クライアントへ提案する**際に使う。
ユーザーがターミナルに貼り付けた**未整形の人材プロフィール**を、提案用の**指定フォーマット（FMT）**へ正確に整形して返す。

案件側を整形する対のスキルが `project-bp-summarize`（こちらは「案件」、本スキルは「人材」）。出力スタイル（コードブロック・pbcopy確認）は揃える。

## 入力

- ユーザーがターミナルに直接貼り付けた、未整形の人材プロフィール（自由形式テキスト）。
- スキルシート（職務経歴書）が **添付ファイル（PDF/Excel等）や Slack/Gmail 添付** で渡されることが多い。その場合は下記「スキルシートの Drive 格納（パートナー人材）」に従い、共有Driveへアップロードして共有URLを `[スキルシート]` に貼る。
- スキルシートのURLが本文に含まれていれば、そのまま `[スキルシート]` に貼る（Drive格納は不要）。
- **1回につき1名のみ**処理する（複数人のベタ貼りは想定しない。複数貼られた場合は1名目のみ整形し、その旨を一言添える）。
- プロフィールが貼られず曖昧な依頼だった場合は、「整形したい人材情報を貼ってください」と促す。

## 変換ルール

1. **提案単価はレンジ化する**（下記「単価変換ロジック」を厳守）。元単価に **+10万 と +15万** の両方を反映し、**円表記**で出力する。
2. **稼働形態は詳細に記載する**。
   - フルリモートの場合 → 「フルリモート（初日出社可）」「フルリモート（月1.2出社可）」など出社条件を括弧で補足する。
   - 一部リモート（出社併用）の場合 → 出力値は「リモート併用（週●日出社可能）」とする。
   - フル出社が可能な場合 → 「常駐」。
   - 入力で稼働形態が特定できなければ、ルール9に従い**行ごと削除**する。
2.5. **稼働日数**は、入力に記載があればその値を記載する。**記載が無い場合は基本的に `週5日` とする**（不明でも削除せず週5日を既定値にする）。
3. **国籍**は、日本人なら `■国籍：` 行を**削除**する。それ以外（韓国籍・中国籍など）の場合はそのまま記載する。
3.5. **最寄り駅 / 住まい**は、駅名があれば `■最寄り駅：●●駅（都道府県）` のように**駅名の右に都道府県を括弧で併記**する（例: `■最寄り駅：目黒駅（東京都）`）。都道府県は、入力（プロフィール本文）に記載があればそれを使い、記載が無ければ駅名から推測して割り出す。**複数の都道府県が考えられて特定できない場合は、無理に括弧を埋めず駅名のみで可**。駅名が無く**住まい（都道府県・エリア）の記載がある場合は、項目名を `■住まい` に変えてその都道府県をそのまま記載**する（例: `■住まい：埼玉県`）。駅名も住まいも無ければ行ごと削除する。
4. **所属**は常に「懇意先個人事業主 / 懇意先正社員」のいずれかを固定表示する。入力でどちらか特定できればその値に絞り、**特定できなければ既定で「懇意先個人事業主」とする**（併記のまま残さない。所属はコア項目として常に残す）。
5. **[概要]** は、経験スキルを漏れなく整理し、わかりやすく箇条書きでまとめる。
6. **[経験スキル]** は、文章中の技術・ツール・フレームワークを **絶対に漏れなく** 拾い、`サーバーサイド／フロントエンド／DB／インフラ／その他` に分類する。情報が無い分類はその行を削除する。1分類でも情報があればその分類は残す。
   - **分類の指針**: gRPC・GraphQL などの通信・API技術、CI/CD・監視・IaC補助ツール・データ基盤系（GitHub Actions, Prometheus, Loki, dbt, Datadog, OpenSearch 等）は **`その他`** に入れる。Vue.js・AngularJS など画面側のFWは **`フロントエンド`** に入れる。
   - **表記ルール（重要）**: 各分類は **`分類：値 / 値 / 値` と同一行に記載**し、値ごとに `・` で改行しない（全角コロン `：` の直後にそのまま値を並べる）。
     - 例: `サーバーサイド：Rust(Actix Web) / Go / Python(Flask/FastAPI) / Ruby(Rails) / Java / Scala / Solidity / NestJS`
     - **例外**: 1行が極端に長くなる／値が非常に多い場合のみ、`・` で改行して箇条書きにしてよい。
   - **FWの表記**: フレームワークは付随する言語に **半角カッコ `()`** で付ける。カッコ内に複数FWがある場合は **スペースを入れず `/` で区切る**。
     - 例: `Python(Flask/FastAPI)` / `Ruby(Rails)` / `Rust(Actix Web)`（"Actix Web" のように単一FW名内のスペースは保持）。
7. **経験職種・担当工程** は、文章に **無いものを追加しない**。記載がある場合のみ表示し、記載が無ければ行ごと削除する。
8. **希望案件・NG案件** は FMT に含めない。
9. **不明な項目（コア項目以外）は行ごと削除する**。情報が無い項目に `●` を残さず、その `■項目：` 行ごと出力から削る。**推測で埋めない**。
   - **例外（コア項目・不明時のみ `●` を残す）**: `提案単価`・`■イニシャル`・`[スキルシート]`・`[概要]`・`[経験スキル]`・`■所属`。
10. **[備考]** は、特記事項・重要な補足事項を箇条書きで記載する。記載事項が **ある場合のみ** FMT 末尾に表示し、無ければ削除する。
11. 提案資料やドキュメントファイルとして出力せず、**チャットベース（コードブロック形式）**で返信する。「フォーマットに沿って提案資料を作成しました」のような定型表現は**使わない**。

## 単価変換ロジック（ルール1の詳細）

提示単価に **下限側は+10万円、上限側は+15万円** を加えてレンジ化し、**円表記**で記載する。

- **単一値で提示**: `X万` → 下限 = X+10、上限 = X+15
  - 例: 80万 → `提案単価：￥900,000 ~ ￥950,000 (140-180h)`
  - 例: 70万 → `提案単価：￥800,000 ~ ￥850,000 (140-180h)`
- **レンジで提示**: `下限L 〜 上限H 万` → 下限 = L+10、上限 = H+15
  - 例: 70〜80万 → `提案単価：￥800,000 ~ ￥950,000 (140-180h)`
- **万→円換算**: 例 90万 → `￥900,000`（カンマ区切り）。稼働時間表記 `(140-180h)` は固定で常に付ける。
- **税込/税別の区別はしない**。提示値をそのまま加算に使う（÷1.1 等の換算はしない）。
- **単価の記載が無い場合**は計算せず `提案単価：￥● ~ ￥● (140-180h)` のまま残す（コア項目）。

## スキルシートの Drive 格納（パートナー人材）

パートナー人材のスキルシートが **添付ファイル（PDF/Excel等）／Slack・Gmail 添付** で渡された場合は、そのファイルを共有Driveへアップロードし、**全員（リンクを知る人）が閲覧可能**な状態にした共有URLを `[スキルシート]` に記載する。本文に既にスキルシートURLがある場合はこの手順は不要。

- **格納先フォルダ（固定）**: `1Z96YZT4gGyguzdTJMlB2lvb0Fh4fG4Cn`
  （https://drive.google.com/drive/u/0/folders/1Z96YZT4gGyguzdTJMlB2lvb0Fh4fG4Cn ）
- **ファイル名の命名規則**: `{イニシャル}{最寄駅名}_スキルシート`
  - イニシャルは **ドットなし**（`O.M` → `OM`）。最寄駅名は **「駅」を除いた駅名**（`原木中山駅` → `原木中山`）。
  - 例: `OM原木中山_スキルシート` ／ `IK守谷_スキルシート`
  - **最寄り駅が無い場合は都道府県名を使う**: 例 `OM千葉_スキルシート`
- **共有権限**: アップロード後に `anyone / reader`（リンクを知る全員が閲覧可）を付与する。
- **取得した共有URL（`webViewLink`）を `[スキルシート]` に貼る**。

WebFetch や gws CLI は使わず、`~/.claude/snippets` の `google_workspace` ユーティリティで Python から直接アップロードする:

```python
import sys
sys.path.insert(0, __import__("os").path.expanduser("~/.claude/snippets"))
from google_workspace import get_drive_service
from googleapiclient.http import MediaFileUpload

svc = get_drive_service()
FOLDER = '1Z96YZT4gGyguzdTJMlB2lvb0Fh4fG4Cn'
path = 'input/career1811010455.pdf'        # スキルシート実ファイル
name = 'OM原木中山_スキルシート'            # {イニシャル}{最寄駅名}_スキルシート

f = svc.files().create(
    body={'name': name, 'parents': [FOLDER]},
    media_body=MediaFileUpload(path, mimetype='application/pdf'),
    fields='id,name,webViewLink', supportsAllDrives=True).execute()
svc.permissions().create(fileId=f['id'], body={'type': 'anyone', 'role': 'reader'},
                         supportsAllDrives=True).execute()
print(f['webViewLink'])   # → [スキルシート] に貼る
```

- mimetype は拡張子に合わせる（Excel は `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`）。
- アップロードできない／実ファイルが手元に無い場合は `[スキルシート]` を `●` のまま残し、出力後にその旨を明示する。

### FMT に関する補足
- `■所属`・`■性別` のように**選択肢が併記されている項目**は、入力で特定できればその値だけに絞る。`■所属` は特定できなければ既定で「懇意先個人事業主」とする（コア項目・併記のまま残さない）。`■性別` は特定できなければルール9に従い行ごと削除する。
- `[スキルシート]` は、入力にURLがあればその行（`●`）にURLを貼る。スキルシートが添付ファイルで渡された場合は「スキルシートの Drive 格納」に従い共有Driveへ上げて共有URLを貼る。どちらも無ければ `●` のまま残す（コア項目）。
- `[経験スキル]` の各分類（サーバーサイド／フロントエンド／DB／インフラ／その他）は、情報が無い分類を見出しごと削除する。
- `[備考]` は記載事項がある場合のみ残す（ルール10）。

## 出力後

- 整形結果（コードブロック）に続けて、**コア項目で `●` のまま残した点・入力の矛盾／要確認点があればリスト**で簡潔に提示し、「不足情報があれば追記して再整形できます」と添える。
- 単価をレンジ変換した場合は、計算根拠（例: 元80万 → 90〜95万）を1行で補足する。
- **最後に必ず「pbcopy しますか？」と確認する**。承認されたら、整形結果（コードブロックの中身＝区切り線から区切り線までの本文）をクリップボードへコピーする。
  - コピー方法: 整形結果テキストを `pbcopy` に渡す（例: 一時ファイル経由で `pbcopy < file`、または `cat <<'EOF' | pbcopy ... EOF`）。コードブロックのバッククォート（```）自体は含めない。
  - 再整形のたびに最新の結果で再確認する。

## 出力後の提案フロー（2分岐）

pbcopy 確認まで終わったら、**この人材をどう提案するか**を赤木に確認する。提案ルートは大きく2つに分岐する。
赤木から明示の指示（「ATS で出す」「セールスチャンネルに投げて」「畑田さんに確認して」等）があればそれに従い、無ければ「ATS等でエンドクライアントへ直接提案 / Slack #sales で弊社メンバーに確認、どちらにしますか？」と尋ねる。

### 分岐A：ATS 等でエンドクライアントへ直接提案

- 整形済みFMT（pbcopy 済み）を赤木がそのまま ATS 等でエンドクライアントへ提案する。**スキル側の追加処理は不要**。
- 「FMT をコピーしました。ATS 等でご提案ください」と一言添えて終了する。

### 分岐B：Slack #sales で弊社メンバーに確認（社内レビュー）

弊社メンバー（畑田さん）に「提案に値するか」を確認してもらうために #sales へ投稿する。

- **企業名が不明な場合のみ**、Base（FreelanceBase）から引く。`~/.claude/snippets/playwright_freelancebase.py` を使い、案件名（または案件の特徴キーワード）で案件を検索し、ひも付くエンド企業名を取得する（WebFetch / gws CLI は使わない）。
- Base からも特定できなければ、**ターミナルで赤木に都度確認する**（推測で企業名を埋めない）。

**2. マッチング診断情報の補足**
- 既にこの人材×案件のマッチング診断（`sales-matching-score-diagnosis` の成果物）があれば、その **html-share 共有URL** と **簡潔な所感（総合スコア・判定・提案可否・要確認点）** を補足として添える。
- 診断がまだ無ければ、赤木に「マッチング診断を添えますか？」と確認する（必要なら `sales-matching-score-diagnosis` を案内）。診断を添えない場合は補足を省略してよい。

**3. 親メッセージの組成（送信前に必ず赤木へ内容確認）**
- 投稿先チャンネル: **#sales（channel ID: `C08B9CD5PQW`）**
- メンション: **常に @Naoki Hatada（user ID: `U08ADFY9GJW`）** を冒頭に付ける。
- 本文の骨子（この順序）:
  1. 見出し文：「{パートナー名}さんから **{エンド企業名}の{案件名}** に対して人材提案（{イニシャル}／{年齢}）がありました。ご確認のほどよろしくお願いします。」
  2. 補足：マッチング診断の共有URL ＋ 簡潔な所感（総合スコア・判定・提案可否・要確認点）。※企業名が不明なまま赤木の確認で確定した場合はその旨も一言添える。
  3. コードブロック：**もともと来た提案サマリー（パートナー受領のまま・スキルシートURL含む）**。整形済みFMT ではなく原文を載せる（社内メンバーが原文を検証できるようにするため）。
