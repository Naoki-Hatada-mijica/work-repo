## 目的

当社（株式会社mijica、フリーランス特化のITエージェント／SES事業）がパートナー企業から紹介される案件は、商流を抜かれないよう情報が抽象化され、具体的なプロダクト名が伏せられている。
このスキルは、間接的な手がかり（勤務地・事業ドメイン・技術スタック・募集要項の文面）から**案件元（エンド企業＝実際に人材を必要とする事業会社）を推定**し、確信度スコア付きで提示して直接開拓につなげる。
以降、特定対象を一貫して「**エンド企業**」と表記する。

## ワークフロー進捗

このチェックリストをコピーして進捗を追跡する:

```
- [ ] ステップ0: 入力取得・案件分割（複数案件時）
- [ ] ステップ1: 手がかり抽出
- [ ] ステップ2: 検索戦略（精度優先・採用媒体の文面照合）
- [ ] ステップ3: 候補生成 → 除外フィルタ
- [ ] ステップ4: 確信度スコアリング & グルーピング
- [ ] ステップ5: HTML生成 → ブラウザ自動起動
- [ ] ステップ6: セルフチェック
```

## 大原則

- **特定対象はエンド企業（案件元の事業会社）のみ**。アグリゲーション・フリーランスエージェント・SES/受託の中間業者は候補に残さない（[除外ブラックリスト](#除外ブラックリスト)）。
- **確信度は推定**。誤特定リスクを下げるため、各候補に**根拠の明示を必須**とする。根拠が弱ければ「参考」に置く。過大評価しない。
- **同一企業の統合はユーザー主導**。スキルは案件を勝手に統合しない。気づいた共通候補は「注記」に留め、断定しない。
- 調査は**メインセッションで実施**（サブエージェントは WebFetch 権限が無いことがある）。

## 手順

### ステップ0: 入力取得・案件分割

1. 入力を取得する（優先順）:
   - input/ にファイルがあれば読む
   - なければ「案件情報をペーストしてください（複数案件まとめてでも可）」と依頼
2. **複数案件のベタ貼り判定**:
   - 区切り文字が無い前提で、案件境界を推定して分割する（職種・案件名・条件のまとまりで区切る）
   - 分割結果を提示して確認を取る:
     ```
     {N}件の案件として認識しました。これで探索を開始してよいですか？
     1. {案件1の1行要約}
     2. {案件2の1行要約}
     ...
     ```
   - ユーザーが誤分割を訂正できるようにする。**確認後に探索開始**。
3. **同一企業の補足を受け取る**: ユーザーが「案件1と3は同じ企業かも」等の補足を渡したら記録し、当該案件群を横断照合する材料にする（断定はしない）。
4. 各案件について、ステップ1〜5をループ実行する。

### ステップ1: 手がかり抽出

案件文から以下を構造化して抽出する（記載のないものは「記載なし」）:

| 区分 | 抽出内容 |
|------|----------|
| 勤務地 | 勤務地 / 最寄駅 / リモート可否・出社頻度 |
| 事業ドメイン | 業種、toC/toB、サービス内容、規模感、フェーズ（新規/保守/内製化） |
| 技術スタック | 必須スキル / 歓迎スキル（言語・FW・クラウド・DB・特殊ツール） |
| ポジション | 職種、チーム体制、人数、PM/PL/メンバー |
| 条件 | 単価レンジ、契約形態、稼働日数、期間、開始時期 |
| シグネチャ | 特徴的な言い回し・固有表現（文面照合のキーになる一文） |

### ステップ2: 検索戦略（精度優先）

抽出した手がかりを組み合わせて WebSearch → 有力候補は採用媒体を深掘り照合する。

**クエリ設計の例:**
- 特徴的フレーズの完全一致: `"{シグネチャの一文}"`（同一求人が別媒体に出ていれば一発で当たる＝最強）
- 条件の組み合わせ: `{勤務地} {主要技術スタック} {ドメイン} エンジニア 求人 中途`
- 自社開発の絞り込み: `{プロダクト特性} 自社開発 エンジニア 採用`
- 媒体横断: `{技術スタック} {勤務地} site:wantedly.com` / `site:green-japan.com` / `site:herp.careers` 等

**深掘り照合（精度優先）:**
- 有力候補の**採用ページ・求人ページを Playwright で取得し、案件文と求人文面を照合**する。
  - WebFetch が JS 描画で取れない／403・999 の場合に Playwright を使う。
  - Playwright は **`/usr/bin/python3`**（system Python, playwright 同梱）で実行する。pyenv の python には未導入。
- LinkedIn は WebFetch を 999 でブロックするため URL のみ拾う。
- **募集要項の文面が直接一致する求人**が当該企業名義で見つかったら、それが特定の決定打。

### ステップ3: 候補生成 → 除外フィルタ

- 検索でヒットした企業を候補化する。
- **[除外ブラックリスト](#除外ブラックリスト)に該当する企業・ドメインを機械的に弾く**。
- 自社プロダクト／サービスを持たず「客先常駐」「SES」「受託」が主たる企業は中間業者とみなし、エンド扱いしない。
- 除外した件数は HTML フッタに記録する。

### ステップ4: 確信度スコアリング & グルーピング

各候補に 0〜100% の確信度を付与する（Claude 本体の定性判断）。判断軸は強い順:

1. **募集要項の文面一致** — 同一文・同一条件の求人が当該企業名義で別媒体に存在（決定打）
2. **技術スタックの一致度** — 必須/歓迎スキルの組み合わせ、特殊技術の一致
3. **事業ドメイン・プロダクト特性の一致**
4. **勤務地・最寄駅の一致**
5. **単価レンジ・契約条件・ポジション・フェーズの整合**

- 複数軸が独立に一致するほど確信度を上げる。単一軸のみの一致は参考止まり。
- 各候補に**根拠を文章で明示**（どの手がかりがどう一致したか、照合した求人URL）。
- グルーピングして全件提示する:
  - **有力**: 確信度 ≥ 65%
  - **可能性あり**: 35〜64%
  - **参考**: < 35%
- 各グループ内は確信度降順。**特定不能なら無理に候補を作らず、「特定不能・追加で欲しい情報」を提示する**。

### ステップ5: HTML生成 → ブラウザ自動起動

- 出力先:
  - 単一案件: `02_task/{タスクフォルダ}/output/{タスクフォルダ名}_結果.html`
  - 複数案件: 案件ごとに `02_task/{タスクフォルダ}/output/{タスクフォルダ名}_案件{n}_{ラベル}.html`（ラベルは要約から簡潔に生成）
- [HTML仕様](#html仕様)に従い生成する。
- **クロス参照注記**: 複数案件で同一企業が候補に挙がった場合、各 HTML に「案件{m}でも企業Zが候補」と注記（断定せずユーザー判断を仰ぐ）。
- 生成後 `open "{ファイルパス}"` でブラウザを自動起動する（複数ファイルはまとめて `open` する）。

### ステップ6: セルフチェック

出力前に以下を確認する:

- [ ] 候補にアグリゲーション・エージェント・SES中間業者が混入していないか
- [ ] 各候補に確信度スコア（%）と**根拠**が記載されているか
- [ ] 企業URL・採用媒体URL が記載されているか（取得不能なら明記）
- [ ] 複数案件のとき、案件ごとに HTML が生成されているか
- [ ] 同一企業の共通ヒットは「注記」に留め、勝手に統合していないか
- [ ] HTML がブラウザで自動的に開いたか

## 除外ブラックリスト

エンド企業以外は除外する。代表例（随時追記）:

**アグリゲーションメディア:**
- フリーランススタート（freelance-start.com）
- フリーランスボード（freelance-board.com）
- フリーランスHub（freelance-hub.jp）
- フリパラ、Frienet 等の案件アグリゲーター

**フリーランスエージェント:**
- レバテックフリーランス（freelance.levtech.jp）
- ギークスジョブ（geechs-job.com）
- Midworks（mid-works.com）
- PE-BANK（pe-bank.jp）
- テクフリ（freelance.techcareer.jp）
- フォスターフリーランス（freelance.fosternet.co.jp）
- クラウドテック（crowdtech.jp）
- ITプロパートナーズ（itpropartners.com）
- ハイパフォーマー・コンサルタント、BIGDATA NAVI 等

**SES・受託の中間業者:**
- 自社プロダクト／サービスを持たず「客先常駐」「SES」「受託開発」が主たる企業
- 多重請けの中間 SES・受託仲介

> 判定補助: 公式サイトを確認し、自社サービス／プロダクトが事業の中心であればエンド候補。求人が「客先常駐」「SES」中心ならエンド扱いしない。

## HTML仕様

**[template.html](template.html) を雛形として使う**。同梱のテンプレートを読み込み、`{{...}}` プレースホルダと冒頭コメントのカード雛形に実データを差し込んで生成する（CSS・レイアウトはテンプレート側で確定済み。体裁の再現性を担保するため、構造は変えない）。

**差し込むプレースホルダ:**
- `{{CASE_LABEL}}` 案件ラベル / `{{GENERATED_AT}}` 生成日時
- `{{CLUE_ROWS}}` 手がかりサマリの表行（`<tr><th>区分</th><td>内容</td></tr>`）
- `{{STRONG_CARDS}}` / `{{POSSIBLE_CARDS}}` / `{{REFERENCE_CARDS}}` 各グループのカード群（テンプレ冒頭のカード雛形を使う。色クラスは `strong` / `possible` / `reference`、バー幅は確信度% を `width` に入れる）
- `{{CROSS_REF}}` クロス参照注記（該当時のみ `<div class="cross-ref">…</div>`、無ければ空文字）
- `{{EXCLUDED_NOTE}}` 除外件数（アグリゲーション/エージェント/SES）・特定不能時の補足

**注意:**
- 各カードに企業名（公式サイトURLリンク）・確信度スコア・事業概要・根拠（一致手がかり＋照合した求人媒体URL）を必ず含める。
- 候補が無いグループは雛形のカードを置かず `<p class="empty">該当なし</p>` を入れる。
- 体裁を崩したい特別な事情がある場合のみ構造を変える。原則テンプレートに従う。
