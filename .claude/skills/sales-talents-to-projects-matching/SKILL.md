## 概要

パートナー企業から提案された人材（1名〜複数名）を起点に、株式会社mijicaが**現在オープンしている案件群**（案件マスタ Doc 全件）へ逆引きで照合する。各人材ごとに最も適合する弊社案件を特定し、その「人材 × 最適案件」の組み合わせを事実ベース・定量優先で 100 点満点に採点する。マッチする案件がない人材は「該当案件なし」と明示する。ランキング＋人材別フル評価（①〜⑥）＋仕分け（提案推奨／見送り／該当案件なし）の Markdown／HTML レポートを `output/` に出力する。

### 姉妹スキルとの使い分け（重要）

| | このスキル `sales-talents-to-projects-matching` | `sales-matching-score-diagnosis` |
|---|---|---|
| 案件の入手 | **弊社オープン案件マスタ Doc 全件**から自動取得 | **ユーザーが提示した案件のみ**（貼付 / input / Slack / Gmail） |
| 向き | 人材起点（N人材 → マスタ全件を逆引き検索） | 双方向（1案件×N候補者／1候補者×N案件・提示分） |
| 典型トリガー | 「営業開始人材が来た、うちのオープン案件で動かせる人は？」 | 「この案件に誰を提案？」「この人をこの2案件に当てると？」 |

**弊社オープン案件マスタ全件から探す**なら本スキル。**提示された案件・候補者だけを直接照合**するなら（向きを問わず）`sales-matching-score-diagnosis` を使う。

## 前提

- 対象業務: SES／業務委託案件の人材紹介（mijica 側が案件保有、パートナー企業が人材提案）
- 1名でも複数名でも一括処理する（複数名の一括逆引きが主ユースケース）
- 評価は Claude 本体に実行させる（Python スクリプトはテキスト抽出・シート読取のみ担当）
- 評価は事実ベースのみ。推測・曖昧表現は禁止
- **スクリプトは複製しない**。テキスト抽出・HTML化・Google API は姉妹スキル／グローバルスニペットを参照する:
  - テキスト抽出: `.claude/skills/sales-matching-score-diagnosis/scripts/extract.py`
  - HTML化: `.claude/skills/sales-matching-score-diagnosis/scripts/md_to_html.py`
  - Google Workspace: `~/.claude/snippets/google_workspace.py`

## 入力の形態

人材情報を以下のいずれか、または組み合わせで受け取る（**案件側はユーザーから渡されない**。案件マスタ Doc から自動取得する）:

1. **Excel ファイル（`.xlsx`）** — `input/` 配下
   - 多くは「1行 = 1人材」のリスト形式。ヘッダ行（氏名/ID/年齢/最寄駅/スキル/単価/稼働開始 等）を解釈し、各データ行を1人材として切り出す
   - 人材ごとにシートが分かれている場合は、各シートを1人材として扱う
2. **Google スプレッドシート** — シート URL または ID
   - `get_sheet_values` で行を取得し、Excel と同様「1行 = 1人材」または列構成を解釈して人材を分割する
3. **Word / PDF（`.docx` / `.pdf`）** — `input/` 配下
   - 1ファイルに複数人材がブロックで並ぶ形式（例: 「①【…】■ID…」「②【…】■ID…」）。区切り（連番・罫線・■ID・イニシャル）で人材ブロックを分割する
4. **Slack スレッド URL / Gmail メッセージID** — MCP 経由で本文＋添付を取得
5. **チャット貼り付けテキスト** — そのまま人材情報として受け取る

**案件マスタ Doc**: https://docs.google.com/document/d/1MthVeWxcFo_l_VFT94xzz4_kKJgf8nWa5XSrbxf1sTM/edit
（Doc ID: `1MthVeWxcFo_l_VFT94xzz4_kKJgf8nWa5XSrbxf1sTM`）

## 手順

### ステップ1: 人材情報の収集

1. PLAN.md と `input/` を確認する
2. ユーザー入力から人材情報のソース（Excel / Google Sheet / docx / PDF / Slack / Gmail / 貼付）を判定する
3. **Google スプレッドシートが指定された場合**:

   ```python
   import sys
   sys.path.insert(0, __import__("os").path.expanduser("~/.claude/snippets"))
   from google_workspace import get_sheet_values
   rows = get_sheet_values('<sheet_id_or_url>', 'A:Z')
   # rows[0] をヘッダとみなし、rows[1:] を各人材として解釈する
   ```

4. **Slack / Gmail の URL・IDが指定された場合**:
   - `mcp__claude_ai_Slack__slack_read_thread` / `slack_read_channel`
   - `mcp__claude_ai_Gmail__gmail_read_message` / `gmail_read_thread`
   - Slack 添付（スキルシートPDF等）は Slack API を直接叩いて `input/` にダウンロードする（`$SLACK_USER_TOKEN` 必須。未設定ならユーザーに依頼し中断）:

     ```bash
     cd "$(pwd)/input" && for fid in F0ABC F0DEF; do
       info=$(curl -s -H "Authorization: Bearer $SLACK_USER_TOKEN" "https://slack.com/api/files.info?file=$fid")
       name=$(echo "$info" | python3 -c "import json,sys;print(json.load(sys.stdin)['file']['name'])")
       url=$(echo "$info" | python3 -c "import json,sys;print(json.load(sys.stdin)['file']['url_private_download'])")
       curl -s -L -H "Authorization: Bearer $SLACK_USER_TOKEN" -o "$name" "$url"
     done
     ```

5. 何名の人材がいるかを確定し、ユーザーに「人材N名を確認しました。弊社オープン案件への逆引きマッチングを行います。」と一言報告してから次へ進む

### ステップ2: テキスト抽出・人材分割

1. `input/` 配下のファイルをテキスト化する:

   ```bash
   python3 .claude/skills/sales-matching-score-diagnosis/scripts/extract.py input/*.pdf input/*.docx input/*.xlsx 2>/dev/null
   ```

2. 抽出結果を**人材単位に分割**する:
   - Excel / Sheet: 1行 = 1人材（または1シート = 1人材）
   - docx / PDF: 連番（①②③）・罫線・「■ID」「■イニシャル」等の区切りで人材ブロックに分割
3. 各人材から以下を内部メモとして抽出する（読み取れない項目は「記載なし」と明示）:
   - 氏名／イニシャル・ID・年齢・性別・最寄駅・国籍
   - 稼働開始時期・希望単価・契約形態
   - スキル（言語/FW/DB/インフラ/工程）・経験年数・役割（PM/PL/SE等）・ドメイン経験
   - スキルシート（経歴書）URL
4. スキャン画像のみの PDF は抽出結果が空になるため、警告を出してユーザーに通知する（OCR は対象外）

### ステップ3: 弊社オープン案件マスタの全件ロード＆構造化

1. 案件マスタ Doc を全件取得する:

   ```python
   import sys
   sys.path.insert(0, __import__("os").path.expanduser("~/.claude/snippets"))
   from google_workspace import get_doc_text
   doc_text = get_doc_text('1MthVeWxcFo_l_VFT94xzz4_kKJgf8nWa5XSrbxf1sTM')
   ```

2. `doc_text` を案件単位に分割し、各案件から以下を構造化する（読み取れない条件は「記載なし」とし減点に使わない）:
   - 案件名（タイトル）・カテゴリ（フロントエンド/Ruby/Go/Python/Node/PM 等）
   - 必須スキル（リスト）／歓迎スキル（リスト）
   - 希望年齢層・単価上限・国籍要件・稼働形態（常駐/リモート）・最寄駅・開始時期
   - ステータスタグ（[🌟新規]/[🔥HOT]/[🙏一時STOP]）。**[🙏一時STOP] は提案過多のため逆引き対象から除外**する
3. ガイドライン上の注意（商流指定・年齢/国籍要件など）も保持する

### ステップ4: 人材ごとの逆引きマッチング（最適案件の特定）

各人材について、ステップ3で構造化した**全オープン案件と照合**し、最も適合する案件を特定する。

- 照合は「必須スキルの重なり」「役割（PM/FE/BE等）の一致」「年齢・単価・国籍・稼働形態の整合」を総合して行う
- **役割の方向性を最優先**する。例: バックエンド専任の人材を、必須中核がフロントエンドの案件に当てても適合は低い（スキルが部分的に重なっても役割不一致は減点要因）
- 上位が拮抗する場合は最大2件まで「最適案件・次点」として保持し、レポートの総合コメントで次点に言及する
- **どの案件にも必須中核が合致しない人材は「該当案件なし」**とし、無理に低スコアの案件を当てない（理由を明示する）
- 単価が人材希望を大きく下回る案件は、スコア上は合致しても商流上現実的でない旨を総合コメントで注記する

### ステップ5: 人材 × 最適案件のスコアリング（①〜⑥）

特定した「人材 × 最適案件」の組み合わせを、以下の配点ルールで採点する。**配点ルールは姉妹スキル `sales-matching-score-diagnosis` と完全に同一**。

#### スコア構成（100点満点）

```
加点小計 = ① 必須条件マッチ度（最大50） + ② 歓迎条件マッチ度（最大20）  ※最大70
正規化得点 = round( 加点小計 × 100 / 70 )                              ※最大100、小数第1位四捨五入
減点小計 = ③ + ④ + ⑤ + ⑥                                              ※最大-30
最終スコア = max(0, 正規化得点 + 減点小計)                              ※下限0
```

| 項目 | 配点（加点） | 配点（減点） |
|------|--------------|--------------|
| ① 必須条件マッチ度 | 最大 +50（正規化前） | — |
| ② 歓迎条件マッチ度 | 最大 +20（正規化前） | — |
| ③ 短期懸念リスク | — | 最大 -10 |
| ④ 国籍リスク | — | 最大 -10 |
| ⑤ 年齢ミスマッチ | — | 最大 -5 |
| ⑥ 金額ミスマッチ | — | 最大 -5 |

- 内訳は **正規化前の加点（XX/50, XX/20）と減点をそのまま** 表示し、「正規化前合計 YY/70 → 正規化後 ZZ/100」を明記する。

#### 基本方針（必須遵守）

- 推測は禁止。案件文・人材情報から読み取れる事実のみを根拠とする
- 曖昧表現は禁止（「やや」「概ね」「そこそこ」等）
- 数値評価には必ず「根拠」を明示
- 確認質問は禁止
- 評価カードは表形式禁止（カードUIのみ使用。表はサマリ／ランキングのみ許容）
- 減点項目は「該当した場合のみ」適用し、③定性評価で必ず言及する

#### 減点ルール

**▼ 短期懸念（最大 -10）**
- 職歴の大半が 6〜12ヶ月未満で終了している場合：**-5**
- 直近3案件の中に1年以上継続した案件が1つも存在しない場合：**-10**
- 1年以上の継続案件が存在する場合：減点なし
- 職歴の継続年数が読み取れない場合：「読み取れない」と明示し、減点しない
- ※ フリーランス／業務委託である点は考慮するが、事実としての継続年数は厳密に評価する

**▼ 国籍（最大 -10）** ※案件に「外国籍不可」等の記載がある場合のみ
- 人材が外国籍：**-10** ／ 帰化済みだが外国籍出身と読み取れる：**-5** ／ 日本国籍のみ：減点なし ／ 国籍不明：減点しない

**▼ 年齢（最大 -5）** ※案件に希望年齢層が明記されている場合のみ
- 希望年齢層から外れている：**-5** ／ 範囲内または情報なし：減点なし

**▼ 金額（最大 -5）**
- 希望単価が案件単価上限を超過：**-5** ／ 範囲内または記載なし：減点なし

#### 判定ラベル

- 90〜100点：★ ベストマッチ（積極提案すべき）
- 80〜89点：★ マッチ度高い（提案して良い）
- 70〜79点：★ 一定マッチ（提案検討。単価・役割の調整論点を明示）
- 60〜69点：★ 低マッチ（基本提案しないがニッチ案件なら可）
- 59点以下：★ マッチ低（提案非推奨）
- 該当案件なし：弊社オープン案件に必須中核が合致せず（スコアは付けない）

### ステップ6: ランキング生成＆仕分け

全人材を「最良マッチスコア」降順で並べ、ランキングサマリを作る（この表のみ表形式を許容）。「該当案件なし」の人材は最下部にまとめる:

```markdown
| 順位 | 人材名 | 最適案件 | スコア | 判定 |
|------|--------|----------|--------|------|
```

さらに以下に仕分ける:
- **提案推奨**（70点以上を目安。役割が合致し提案価値がある人材）
- **見送り寄り**（60点未満／役割ミスマッチ）
- **該当案件なし**

### ステップ7: レポート出力

[REPORT_TEMPLATE.md](REPORT_TEMPLATE.md) に沿って、以下を `output/{ディレクトリ名}_逆引きマッチングレポート.md` に書き出す:

1. 結論サマリ（提案推奨／見送り／該当案件なしの仕分け）
2. ランキングサマリ（表）
3. 人材別フル評価（①〜⑥。当てた案件名を明記）

**出力後の必須アクション:**

1. **ターミナルに全文出力**: ファイル書き込み後、レポート全文をアシスタント応答内にも貼り付けて表示する
2. **Markdown 版は書き出すのみ（自動オープンしない）**
3. **HTML 版を並行出力 + ブラウザで自動オープン**（許可不要・自動実行）:

   ```bash
   html_path=$(python3 .claude/skills/sales-matching-score-diagnosis/scripts/md_to_html.py "output/{ディレクトリ名}_逆引きマッチングレポート.md") \
     && open "$html_path"
   ```

   - `markdown` 未導入で失敗した場合は `pip3 install markdown` 後に再試行
   - HTML 生成または `open` に失敗した場合はタスク完了扱いとし、「HTML を開けませんでした」と一言報告する

**人材別フル評価の出力フォーマット（厳守）:**

```markdown
### 人材N: {氏名／イニシャル}（{年齢}・{推定スキル像}）

**当てた案件：{案件名}（単価{…}・希望年齢{…}・{稼働形態}）**
※（役割が噛み合わない場合はその旨を1行で注記）

# 🎯 ① トータルスコア
- 合計スコア：XX点（100点満点）
- 判定：★ {ラベル}
- 内訳：① XX/50 ② XX/20 ③ -X ④ -X ⑤ -X ⑥ -X
- 計算: 加点 YY/70 → 正規化 ZZ/100 → 減点 -W → **最終 XX点**

---

# 📊 ② 定量評価

## ■ 必須条件マッチ度（平均 XX%）

🟦 **要件**：{案件の必須要件}
🔍 **根拠**：{人材情報の事実}
📈 **マッチ度**：XX%

（必須要件すべて同形式）

## ■ 歓迎条件マッチ度（平均 XX%）

🟩 **要件**：{案件の歓迎要件}
🔍 **根拠**：{人材情報の事実}
📈 **マッチ度**：XX%

---

# 📝 ③ 定性評価

## ● 強み
- {事実ベース}

## ● 弱み
- {事実ベース}

## ● 案件への影響
- {減点項目・役割ミスマッチがある場合は必ず言及}

---

# 🌐 ④ 総合コメント（200〜300文字）

{統合評価。次点案件があれば言及。単価・役割の調整論点を明示。主観・感情表現禁止。}

---

# 📨 ⑤ クライアント向け推薦文（100〜150文字）

{ビジネス文体。改行1箇所のみ。}

---

# 📨 ⑥ 紹介元パートナー企業宛 見送り連絡（提案非推奨＝69点以下／該当案件なし のみ）

{100〜150文字。改行1箇所のみ。スキル軸が合致しない理由を明確に。宛先は紹介元パートナー企業。}
```

**「該当案件なし」人材の出力**: ①〜②のスコアカードは省略し、「### 人材N: …（該当案件なし）」見出しの下に③定性評価（軸の不一致理由）・④総合コメント・⑥見送り連絡のみを出力する。

**⑥ の出力ルール（重要）:**
- **提案非推奨（69点以下）または「該当案件なし」**の人材のみ出力する
- **70点以上**の提案推奨人材では出力しない（提案するため見送り連絡は不要。代わりに④⑤で提案論点を明示）
- 宛先は「紹介元パートナー企業」であり、人材本人宛ではない

**セルフチェック（出力後に必ず実施）:**

1. ランキング表の順序がスコア降順と一致し、「該当案件なし」が最下部にまとまっているか
2. ⑥見送り連絡が70点以上の人材に含まれていないか
3. ⑥見送り連絡が69点以下／該当案件なしの人材に必ず含まれているか
4. 評価カードに表形式が混入していないか（サマリ・ランキング以外）
5. 減点項目・役割ミスマッチが③定性評価で言及されているか
6. 当てた案件が [🙏一時STOP] 案件でないか

1つでも違反があれば修正して再出力する。

### ステップ8: サマリ報告

レポート全文をターミナルに出力した**後**、末尾に以下のサマリを追記する:

- 評価した人材数
- 提案推奨人材（70点以上）の氏名・最適案件・スコア
- ベストマッチ（90点以上）の有無
- 見送り寄り／該当案件なしの人数
- 出力ファイルパス（Markdown 版・HTML 版の両方）
- HTML をブラウザで開いた旨（`open` 実行成功時）

## 非スコープ

- スキャン画像のみの PDF の OCR
- `.xls`（旧 Excel）は未対応（`.xlsx` に保存し直して投入する）
- Slack / Gmail への自動返信（見送り連絡は文面生成のみ。送信判断は赤木が行う）
- スコア履歴の DB 蓄積
