# ベストプラクティス チェックリスト

監査時にこのチェックリストを上から順に確認する。

## 1. フロントマター

- [ ] `name` が存在する
- [ ] `name` が64文字以下
- [ ] `name` が小文字・数字・ハイフンのみ
- [ ] `name` に予約語（`anthropic`, `claude`）を含まない
- [ ] `description` が存在し空でない
- [ ] `description` が1024文字以下
- [ ] `description` にXMLタグを含まない
- [ ] `description` が三人称で書かれている（「〜します」「〜する」）
  - NG: 「お手伝いします」「これを使って〜できます」
- [ ] `description` にスキルの機能（何をするか）が含まれている
- [ ] `description` に使用トリガー（いつ使うか）が含まれている
- [ ] `description` に主要な用語・キーワードが含まれている

## 2. 命名規則

- [ ] 動名詞形（-ing）または明確な名詞句を使用している
  - 推奨: `processing-pdfs`, `reviewing-contracts`
  - 許容: `pdf-processing`, `contract-review`
  - NG: `helper`, `utils`, `tools`, `documents`

## 3. 簡潔さ

- [ ] SKILL.md のボディ（フロントマター除く）が500行以下
- [ ] Claudeが既知の一般的な概念を冗長に説明していない
  - 例: PDFの説明、ライブラリの基本的な使い方の長い説明
- [ ] 各段落がそのトークンコストに見合う価値を提供している

## 4. 構造と段階的開示

- [ ] 参照ファイルはSKILL.mdから1レベル深いまで（2段以上のネストなし）
  - NG: SKILL.md → advanced.md → details.md
- [ ] 100行以上の参照ファイルには目次がある
- [ ] ファイルパスにフォワードスラッシュを使用（バックスラッシュなし）
- [ ] ファイル名が内容を示す説明的な名前になっている
  - NG: `doc1.md`, `file2.md`
- [ ] 大きなコンテンツは別ファイルに分離し、SKILL.mdからリンクしている

## 5. ワークフロー

- [ ] 複数ステップのタスクにはチェックリストがある
- [ ] ステップに明確な順序がある
- [ ] フィードバックループがある（検証→修正→再検証）
  - 特にファイル操作・外部出力を伴うスキルで重要
- [ ] 条件分岐がある場合、明確なフロー指示がある

## 6. コンテンツ品質

- [ ] 時間に敏感な情報がない
  - NG: 「2025年8月以降は〜」
  - OK: 「レガシーパターン」セクションに旧情報を格納
- [ ] 用語が一貫している（同じ概念に複数の呼び方を使っていない）
- [ ] デフォルトの推奨を1つ示し、代替は条件付きで記載
  - NG: 「AまたはBまたはCを使えます」
  - OK: 「Aを使用。スキャンPDFの場合はBを使用」
- [ ] 例が抽象的でなく具体的

## 7. スクリプト付きスキル（該当する場合のみ）

- [ ] エラーハンドリングが明示的（Claudeに任せていない）
- [ ] マジックナンバーがない（定数に理由がコメントされている）
- [ ] 必要なパッケージと依存関係が明記されている
- [ ] スクリプトの実行方法が明確（実行 vs 参照読みの区別）
- [ ] 検証ステップがある（バリデーション → 修正 → 再実行）
- [ ] MCPツール参照は完全修飾名を使用（`ServerName:tool_name`）

## 8. 自由度の設定

- [ ] タスクの脆弱性に応じた自由度が設定されている
  - 脆弱な操作（DB移行等）: 低い自由度（具体的なスクリプト）
  - 柔軟なタスク（コードレビュー等）: 高い自由度（ヒューリスティック）
- [ ] テンプレートの厳密さがニーズに合っている
