## 概要

株式会社mijicaの案件と候補者の適合度を、事実ベース・定量優先で 100 点満点に採点する。**固定側（1件）× 比較側（複数）** を照合し、比較側をスコア降順でランキングする。ランキングサマリ＋比較側ごとのフル評価（①〜⑥）＋紹介元パートナー別の「まとめ見送り連絡」を Markdown／HTML レポートとして `output/` に出力する。

評価軸（必須/歓迎条件は案件から抽出し、候補者の事実を根拠にマッチ度を算出）は方向に依らず同一。**どちらを 1件に固定するか** だけが異なる。

### 姉妹スキルとの使い分け（重要）

| | このスキル `sales-matching-score-diagnosis` | `sales-talents-to-projects-matching` |
|---|---|---|
| 入力 | **案件・候補者ともユーザーが提示**（貼付 / input / Slack / Gmail） | 人材のみ提示。**案件は案件マスタDoc全件**から自動取得 |
| 向き | 双方向：1案件×N候補者／1候補者×N案件（提示分のみ） | 人材起点：N人材 → オープン案件マスタ全件を逆引き検索 |
| 典型トリガー | 「この案件に誰を提案？」「この人をこの2案件に当てると？」 | 「営業開始人材が来た、うちのオープン案件で動かせるのは？」 |

**提示された案件だけを対象にするなら本スキル。** 弊社オープン案件マスタ全件から探すなら `sales-talents-to-projects-matching`。

## 対応モード（重要）

入力の構成からモードを判定する。用語は **固定側（アンカー）= 1件に固定する側**、**比較側（バリアント）= 複数で比較・ランキングする側**。

| モード | 固定側 | 比較側 | ランキング対象 |
|--------|--------|--------|----------------|
| **案件起点（1:N）** | 案件 1件 | 候補者 N名 | 候補者 |
| **人材起点（N:1）** | 候補者 1名 | 案件 N件（提示分） | 案件 |

- スコアは常に「案件の条件 × 候補者の事実」のペアで算出する（固定側がどちらでも計算式は不変）。
- **N:N（複数案件 × 複数人材の総当たり）は対象外。** 両側とも複数で渡された場合は、どちらを固定側にするかをユーザーに確認してから 1:N もしくは N:1 に落とす。

## mijica の立場（重要）

⑤推薦文・⑥見送り連絡の **宛先と文面の向き** が変わるため、mijica がどちら側かを必ず確定させる。

| mijica の立場 | 状況 | ⑤推薦文の宛先 | ⑥見送り連絡の宛先 |
|---------------|------|----------------|--------------------|
| **案件側（既定）** | mijica が案件を保有し、パートナーが人材を提案 | mijica のクライアント（エンド）向け | **人材を提案してくれたパートナー**（候補者をお断り） |
| **人材側** | mijica が人材を保有し、パートナーの案件へ提案 | **案件を出したパートナー**（案件元）向け | 案件を出したパートナー（エントリー辞退） |

- 入力から立場が判別できない場合は **推測せず、必ずユーザーに確認する**（ステップ0）。判別の手掛かり: 「御社配信の案件」=mijicaが案件側／「弊社の人材をご提案」=mijicaが人材側／案件票の出所・人材の所属（"弊社直"が誰を指すか）など。

## 手順

### ステップ0: モード判定と mijica 立場の確認

1. 入力構成からモードを判定する:
   - 案件 1件 ＋ 候補者 複数 → **案件起点（1:N）**
   - 候補者 1名 ＋ 案件 複数 → **人材起点（N:1）**
   - 案件 1件 ＋ 候補者 1名 → どちらの固定でも同一（1×1）。そのまま 1 ペアで評価
   - 両側とも複数 → N:N は対象外。どちらを固定側にするかユーザーに確認
2. **mijica の立場を確定する**（「mijica の立場」表参照）。入力から判別できない場合は次のように確認する:
   > 「今回 mijica は **案件側（案件を保有しパートナーが人材提案）／人材側（mijica の人材をパートナー案件へ提案）** のどちらでしょうか。見送り・推薦文の宛先が変わるため確認させてください。」
3. 判定結果（モード・固定側・比較側・mijica立場）を一言報告してから次へ進む。

### ステップ1: 入力の収集

1. PLAN.md と `input/` を確認する
2. **案件情報がタイトルのみのとき**は案件マスタ Doc から本文を引き当てる:

   ```python
   import sys
   sys.path.insert(0, __import__("os").path.expanduser("~/.claude/snippets"))
   from google_workspace import get_doc_text
   doc_text = get_doc_text('1MthVeWxcFo_l_VFT94xzz4_kKJgf8nWa5XSrbxf1sTM')
   ```

   - 完全一致 → 部分一致 → 表記ゆれ（全角/半角・スペース・英数字大小・記号差）の順でフェイルオーバー
   - 複数ヒット時は一覧を提示して確認
   - 案件票が全文で提示されている場合は Doc 検索をスキップしそのまま使う
   - **案件マスタ Doc で全くマッチしないときは、中断せず FreelanceBase 検索フォールバック（下記 6 番）へ進む**（FB でも見つからなければ中断。推測で類似案件を採用しない）

3. **Slack / Gmail の URL・ID** が指定されていれば本文を取得する:
   - `mcp__claude_ai_Slack__slack_read_thread` / `slack_read_channel`
   - `mcp__claude_ai_Gmail__gmail_read_message` / `gmail_read_thread`
   - 本文には案件と候補者が混在しやすい。件名/冒頭行/「案件名:」「【案件】」「ご提案」「ご紹介」等のラベルから **案件タイトル候補** と **候補者プロフィール候補** をそれぞれ切り出す。氏名・年齢・所属・スキル・希望単価・最寄駅・国籍の羅列＝候補者。添付名に「スキルシート/経歴書/職務経歴/SS/CV」を含むもの＝候補者添付
   - 抽出に成功したら「本文から案件『XXX』と候補者『YYY』を抽出しました」と報告。曖昧なら候補一覧を提示して確認
   - **差出人/投稿者の所属（パートナー企業名）を控える**（見送りグルーピング用）

4. **Slack 添付ファイル（職務経歴書PDF等）の取得** — Slack MCP はDLできないため Slack API を直接叩く:
   - 事前準備（初回）: Slack App に `files:read` User Token Scope を付与し `~/.zshrc` に `export SLACK_USER_TOKEN=xoxp-...`。未設定ならユーザーに依頼し中断
   - `slack_read_thread` の `Files: {name} (ID: {file_id}, ...)` から file_id を取得し一括DL:

     ```bash
     cd "$(pwd)/input" && for fid in F0ABC F0DEF; do
       info=$(curl -s -H "Authorization: Bearer $SLACK_USER_TOKEN" "https://slack.com/api/files.info?file=$fid")
       name=$(echo "$info" | python3 -c "import json,sys;print(json.load(sys.stdin)['file']['name'])")
       url=$(echo "$info" | python3 -c "import json,sys;print(json.load(sys.stdin)['file']['url_private_download'])")
       curl -s -L -H "Authorization: Bearer $SLACK_USER_TOKEN" -o "$name" "$url"
     done
     ```

   - Gmail 添付は MCP で直接取得可能（別手順不要）

5. **スキルシートが Google Sheet / Drive 上の Excel のとき**:
   - ネイティブ Sheet なら `get_sheet_values(<url>, 'A1:Z200')`
   - **Drive 上の Excel（Office ファイル）は Sheets API が「This operation is not supported... must not be an Office file」で失敗する。** その場合は Drive API で `get_media` ダウンロードしてから `extract.py` に渡す。**共有ドライブ／共有リンクのファイルは `supportsAllDrives=True` を付けないと 404 になる**:

     ```python
     import sys, io
     sys.path.insert(0, __import__("os").path.expanduser("~/.claude/snippets"))
     from google_workspace import get_drive_service
     from googleapiclient.http import MediaIoBaseDownload
     svc = get_drive_service(); fid = '<file_id>'
     meta = svc.files().get(fileId=fid, fields='name,mimeType', supportsAllDrives=True).execute()
     req = svc.files().get_media(fileId=fid, supportsAllDrives=True)
     fh = io.FileIO('input/skillsheet.xlsx', 'wb'); dl = MediaIoBaseDownload(fh, req)
     done = False
     while not done: _, done = dl.next_chunk()
     fh.close()
     ```

6. **FreelanceBase 検索フォールバック（案件・人材とも）** — 案件がマスタ Doc に、人材が input/Slack/Gmail/貼付に **見つからなかったときのみ** 発火させる。提示・シート・添付がある側はこのフォールバックを使わない。

   ```bash
   # 案件をタイトル/キーワードで検索（一致案件をフル整形 JSON で出力）
   python3 .claude/skills/sales-matching-score-diagnosis/scripts/fb_search.py jobs "インフラ基盤構築支援"
   # 人材を氏名/スキル等で検索（生 JSON で出力）
   python3 .claude/skills/sales-matching-score-diagnosis/scripts/fb_search.py candidates "山田" --limit 10
   ```

   - **OTP 不要・env 認証**（`FREELANCEBASE_EMAIL` / `FREELANCEBASE_PASSWORD`）。セッションは `~/.cache/sales-matching/fb_state.json` に再利用キャッシュされる
   - 引き当ては 完全一致 → 部分一致 → 表記ゆれ の順。複数ヒット時は一覧を提示して確認、0 件なら中断（推測で類似を採用しない）
   - **案件 JSON の主要フィールド**: `title`(=name) / `detail`(案件内容) / `required`(必須) / `welcome`(歓迎) / `skill_desc`(開発環境) / `monthly_price_from`〜`to`(仕入れ単価・円/月) / `expected_age_min`〜`max`(希望年齢) / `accept_foreigner_type_id`(国籍要件) / `inception_day`(開始) / `work_style` / `business_day` / `company`(エンド企業) / `skills` / `occupation`

   - **人材 JSON** は項目が多い生レコード。氏名・所属・スキル・希望単価・年齢・最寄駅・国籍等を読み取り、ステップ3 の候補者構造化に使う。読み取れない項目は「記載なし」とする
   - スキルシート/経歴書の本文は FB レコードに含まれないことがある。詳細が必要なら別途 input/Drive 等から取得する

### ステップ3: 案件条件の構造化（固定側・比較側の整理）

- **各案件**から以下を抽出して内部メモに保持する（人材起点モードでは案件が複数あるので案件ごとに）:
  - 必須条件（リスト）／歓迎条件（リスト）
  - 希望年齢層（記載があれば）／単価上限（記載があれば）／国籍要件（「外国籍不可」等があれば）
  - 稼働形態・開始時期・支払サイト・精算幅（減点対象外でも定性評価で使う）
- **各候補者**から以下を抽出する（読み取れない項目は「記載なし」と明示）:
  - 氏名/イニシャル・年齢・国籍・最寄駅・所属/契約形態・希望単価・稼働開始時期
  - スキル（言語/FW/DB/インフラ/工程）・経験年数・役割（PM/PL/SE等）・職歴の継続年数・ドメイン経験
- **役割・業務詳細を時系列で精査する（必須）**: 技術キーワードの有無だけで判定しない
  - 各案件の役割欄から PM/PL/PMO 専任期間を実装経験から除外し、**直近の実装ブランク**を算出する
  - 「経験◯年」は営業職等の非エンジニア期間を含む通算でないかを確認する
  - 紹介元の営業文（推薦文）の経験年数・技術構成は**スキルシート本体と突合**し、不一致はレポートの弱みに明記する。営業文由来の盛り数字（経験◯年等）をそのまま推薦文に使わない
- 読み取れない条件は「記載なし」とし、後段の減点には使わない。

### ステップ4: スコアリング

固定側 × 各比較側の **ペアごと**に、以下の配点ルールで採点する（案件起点なら各候補者、人材起点なら各案件が1ペア）。

#### スコア構成（100点満点）

加点側（必須50＋歓迎20＝最大70点）を **100点ベースに正規化** し、減点を引いて最終スコアを算出する。

```
加点小計 = ① 必須条件マッチ度（最大50） + ② 歓迎条件マッチ度（最大20）  ※最大70
正規化得点 = round( 加点小計 × 100 / 70 )                              ※最大100、小数第1位四捨五入
減点小計 = ③ + ④ + ⑤ + ⑥                                              ※最大-30
最終スコア = max(0, 正規化得点 + 減点小計)                              ※下限0
```

| 項目 | 配点（加点） | 配点（減点） |
|------|--------------|--------------|
| ① 必須条件マッチ度 | 最大 +50（正規化前） | — |
| ② 歓迎条件マッチ度 | 最大 +20（正規化前） | — |
| ③ 短期懸念リスク | — | 最大 -10 |
| ④ 国籍・出自リスク（国籍＋海外大学卒） | — | 最大 -10 |
| ⑤ 年齢ミスマッチ | — | 最大 -5 |
| ⑥ 金額ミスマッチ | — | 最大 -5 |

- 必須・歓迎ともマッチ度100%、減点ゼロで 100点（70 × 100/70 = 100）。
- 内訳表示は **正規化前の加点（XX/50, XX/20）と減点をそのまま** 表示し、「正規化前合計 YY/70 → 正規化後 ZZ/100」を併記する。
- 必須条件のうち「スキルではない確認事項」（例: アダルトコンテンツ抵抗の有無）で本人情報が無いものは、0%扱いで平均を不当に下げず **「確認要（評価対象外）」** として平均算出から除外し、定性評価で要確認と明示する。

#### 基本方針（必須遵守）

- 推測（無関係領域への飛躍）は禁止。案件文・スキルシートから読み取れる事実を根拠とする。ただし**技術的に強く随伴・含意するスキルの推認は下記「隣接・含意スキルの推認」ルールに従って可**
- 曖昧表現は禁止（「やや」「概ね」「そこそこ」等）
- 数値評価には必ず「根拠」を明示
- スコアリング中の確認質問は禁止（※モード・mijica立場の確認はステップ0で済ませる）
- 評価カードは表形式禁止（サマリ／ランキングのみ表を許容）
- 減点項目は「該当した場合のみ」適用し、③定性評価で必ず言及する
- 役割が PG/PL/PM 併記の案件は、業務詳細の記載から実装比率を判断する。断定できなければマッチ度を下げ、③定性評価の「面談での必須確認事項」に挙げる

#### 隣接・含意スキルの推認（必須遵守）

案件の要求スキルが候補者経歴に**明示されていなくても、実務上ほぼ必ずセットで使われる隣接スキル・含意スキルの実務がある場合は、それを根拠に高いマッチ度を推認する**。「明示記載なし」だけを理由に機械的に減点しない。

- **推認してよい例（技術的に強く随伴する）**:
  - AWS ECS 運用 → **ECR**（コンテナイメージ管理）利用を推認
  - Kubernetes 運用 → kubectl / Helm / コンテナレジストリ利用を推認
  - Terraform 構成のリファクタリング・依存関係整理 → Module 設計・State 管理の実務を推認
  - 長期の監視体制構築実績（Zabbix 等）→ 別ツール（Datadog 等）でも「監視体制を構築する力」は応用可能と推認
  - SLA・非機能要件の策定経験 → SLI/SLO 設計思考への近接として加点
- **推認してはいけない例（無関係領域への飛躍＝従来どおり推測禁止）**:
  - AWS 経験がある → Vector DB（Qdrant 等）のチューニング経験がある（別領域、×）
  - バックエンド言語の経験がある → 特定ドメイン（金融・医療等）の業務知識がある（×）
- **根拠欄に推認である旨を必ず明示する**（例:「ECR は明示なしだが ECS 運用から強く推認」）。事実と推認は区別する
- **ツール特化の深さ（規模・期間）に確認余地が残る場合は満点にせず高め（80〜90% 程度）に留め**、③定性評価で「要確認」と注記する
- 隣接スキルでも補えない **完全に独立した必須要件**（例: インフラ職歴に対する「バックエンド開発実装 5 年以上」）は推認で埋めず、事実ベースで厳密に評価する

#### 減点ルール

**▼ 短期懸念（最大 -10）**
- 職歴の大半が 6〜12ヶ月未満で終了：**-5**
- 直近3案件に1年以上継続した案件が1つも無い：**-10**
- 1年以上の継続案件が存在：減点なし
- 継続年数が読み取れない：「読み取れない」と明示し減点しない
- ※ フリーランス／業務委託の点は考慮するが、事実としての継続年数は厳密に評価する

**▼ 国籍・出自（最大 -10）**

④は「(1) 国籍」と「(2) 海外大学卒業歴」の2シグナルを **共通ティア表** で評価し、両者の減点を **合算**する（合算後、④全体で最大 -10 に丸める）。

**共通ティア表**（出身国／大学所在国で判定。数値が大きいほど大きく減点）:
- **ティアA（最大）＝シンガポール／フィリピン／タイ／ベトナム／インドネシア：-10**
- **ティアB（中）＝ティアA・中国・韓国・米国・英国 以外：-7**
- **ティアC（小）＝中国／韓国：-4**

**(1) 国籍シグナル** ※案件に「外国籍不可」等の記載がある場合のみ評価
- 日本国籍のみ：0 ／ 国籍不明：0（減点しない）
- 外国籍：出身国の共通ティアを適用
- **帰化済みだが外国籍出身と読み取れる**場合：出身ティアの **約半分**（A:-5 ／ B:-3 ／ C:-2）
- 外国籍と判るが出身国を特定できない場合：ティアB（-7）を適用し、③定性評価で「出身要確認」と明示

**(2) 海外大学卒業シグナル** ※案件の外国籍要件に関わらず常に評価（スキルシート信頼性の観点）
- 日本の大学：0
- **米国／英国 の大学：0（無関係＝減点しない）**
- その他の海外大学：大学所在国の共通ティアを適用（ティアB は「ティアA5か国・中国・韓国・米国・英国 以外の海外大学」）
- 海外大学卒と判るが所在国を特定できない場合：ティアB（-7）を適用し、③定性評価で「所在要確認」と明示

**合算例**: 日本国籍(0) ＋ シンガポール大学卒(A:-10) = **-10** ／ 中国国籍(C:-4) ＋ 米国大学卒(0) = **-4** ／ ベトナム国籍(A:-10) ＋ ベトナム大学卒(A:-10) = 合算 -20 → 丸めて **-10**

**▼ 年齢（最大 -5）** ※案件に希望年齢層が明記されている場合のみ
- 範囲外：**-5** ／ 範囲内または情報なし：減点なし

**▼ 金額（最大 -5）**
- 希望単価が案件単価上限を超過：**-5** ／ 範囲内または記載なし：減点なし

#### 判定ラベル

- 90〜100点：★ ベストマッチ（積極提案すべき）
- 80〜89点：★ マッチ度高い（提案して良い）
- 70〜79点：★ 一定マッチ（他候補がいればそちら優先／案件起点では次点）
- 60〜69点：★ 低マッチ（基本提案しないがニッチ案件なら可）
- 59点以下：★ マッチ低（提案非推奨）

レポート全文をターミナルに出力した**後**、末尾に以下を追記する:

- モード（案件起点／人材起点）・固定側・mijica立場
- 評価した比較対象数（候補者数 or 案件数）
- トップ（1〜3位）の名称・スコア・判定
- ベストマッチ（90点以上）の有無
- 見送り対象（79点以下）の件数と、まとめ見送り連絡を作成したパートナー数
- 出力ファイルパス（Markdown 版・HTML 版の両方）
- HTML をブラウザで開いた旨（`open` 実行成功時）

## 非スコープ

- スキャン画像のみの PDF の OCR
- 弊社オープン案件マスタ全件からの逆引き検索（→ `sales-talents-to-projects-matching`）
- N:N（複数案件 × 複数人材の総当たり）マトリクス評価
- Slack / Gmail への自動返信（見送り・推薦文は文面生成のみ。送信判断は赤木が行う）
- スコア履歴の DB 蓄積
