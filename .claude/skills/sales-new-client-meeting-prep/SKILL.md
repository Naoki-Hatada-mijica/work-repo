## 前提

- 株式会社mijicaはフリーランス特化のITエージェント業を営んでいる
- 商談先はフリーランスの紹介先となるクライアント企業
- 調査で見つからない情報は「情報なし」と明記する（推測で埋めない）

## 手順

### ステップ1: 入力の取得

以下の優先順位で入力を取得する:

1. **PLAN.md を読む** — 企業名・担当者名等が記載されていればそちらを使用
2. **ユーザーに確認** — 「企業名と担当者名（その他わかっている情報）をペーストしてください」と依頼

取得すべき情報:
- 企業名（必須）
- 担当者名（任意、あれば調査対象に含める）
- その他メモ（任意）

### ステップ2: 企業の同定

企業名でWebSearchを実行し、該当する企業を特定する。

**同名企業が複数存在する場合:**
- 候補を一覧で提示し、ユーザーに確認する
- 確認後に次のステップへ進む

**一意に特定できた場合:**
- コーポレートサイトのURLを控えて次へ進む

### ステップ3: 情報調査（並列実行）

以下の調査をサブエージェントで**並列実行**する。各サブエージェントには企業名とコーポレートサイトURLを渡す。

#### 3-A: 企業基礎情報 + 資金調達

WebSearchで以下を調査する:
- 会社名、設立年、所在地、代表者、従業員数、資本金
- コーポレートサイトURL
- 直近の資金調達ラウンド・金額・時期
- 主要な投資家・VCの名前
- 資金調達の経緯（シード→シリーズA→...の時系列）

検索キーワード例: `"{企業名}" 資金調達`, `"{企業名}" シリーズ`, `"{企業名}" 会社概要`

#### 3-B: 事業概要 + 競合分析

WebSearchで以下を調査する:

**事業概要:**
- 全事業・プロダクトの一覧と概要
- メイン事業の特定と詳細説明

**競合分析（メイン事業の競合）:**
- 競合企業の一覧
- 各競合の詳細:
  - 事業概要
  - 資金調達状況（直近ラウンド・金額）
  - 従業員規模
  - 差別化ポイント（対象企業との違い）

検索キーワード例: `"{企業名}" 事業`, `"{プロダクト名}" 競合`, `"{プロダクト名}" vs`

#### 3-C: 採用情報

以下の優先順位で求人情報を調査する:

1. **コーポレートサイトの採用ページ** → ATS（HERP、HRMOS、Lever等）へのリンクがあればそちらを優先的に調査
2. **Wantedly**: `"{企業名}" site:wantedly.com` で検索
3. **Green**: `"{企業名}" site:green-japan.com` で検索
4. **その他**: `"{企業名}" エンジニア 業務委託 募集` 等で検索

対象職種: エンジニア、PM、デザイナー、マーケター等のプロダクト開発領域

各求人について以下を記録:
- 職種名
- 雇用形態（**業務委託は優先的にピックアップ**）
- 技術スタック
- 年収/単価（記載あれば）
- 掲載元URL

#### 3-D: 担当者情報

担当者名が提供されている場合のみ実行する。担当者名がない場合、レポートのセクション6（担当者情報）は省略する。

**調査手順:**

1. **一般検索** — `"{担当者名}" "{企業名}"` で経歴・役職・インタビュー記事等を調査
2. **LinkedIn検索** — `"{担当者名}" site:linkedin.com` で検索し、検索結果のスニペット（役職・経歴の断片）とプロフィールURLを取得する
   - LinkedInはWebFetchでのアクセスをブロックするため、ページ内容の直接取得は不可
   - 検索結果のスニペットに含まれる情報（役職、企業名等）を活用する
   - プロフィールURLが見つかった場合は必ずレポートに記載する（ユーザーが商談前に自分で確認できるようにする）
3. **その他SNS検索** — `"{担当者名}" "{企業名}"` に加えて以下を検索:
   - `"{担当者名}" site:x.com OR site:twitter.com`
   - `"{担当者名}" site:note.com`
   - `"{担当者名}" site:speakerdeck.com OR site:slideshare.net`
4. **登壇・メディア検索** — `"{担当者名}" インタビュー OR 登壇 OR セミナー`

**収集する情報:**
- 氏名、職務役職、管轄部門
- キャリア経歴（前職等）
- SNSアカウントURL（LinkedIn、X/Twitter、note等）— リンクの存在自体が価値あるため、内容取得不可でもURLは必ず記載
- 公開されている登壇・インタビュー記事

### ステップ4: レポート生成

[REPORT_TEMPLATE.md](REPORT_TEMPLATE.md) の形式に従い、調査結果を `report.md` としてタスクフォルダ直下に出力する。

- 担当者名がない場合はセクション6（担当者情報）を省略する
- 情報が見つからなかった項目は「情報なし」と明記する

### ステップ5: ユーザー確認・追加調査

レポートの主要な発見事項をサマリとして表示する:
- 業務委託求人の有無と件数
- 担当者の役職・管轄部門
- 特筆すべき資金調達や競合状況

ユーザーに以下を確認する:
- 内容に過不足がないか
- 追加で深掘りしたい項目がないか
- 修正が必要な箇所がないか

修正・追加調査があれば反映してからステップ6へ進む。

### ステップ6: VSCodeで開く

```bash
code {タスクフォルダの絶対パス}/report.md
```

**注意:** `report.md` はタスクフォルダの絶対パスで指定する。

### ステップ7: FreelanceBase へ登録＋レポート投稿

レポート確認後、FreelanceBase の企業一覧に対象企業を登録（既存ならそのまま使用）し、商談相手の担当者を追加、`report.md` 全文を「コメント」タブへ投稿する。

**前提**
- 環境変数 `FREELANCEBASE_EMAIL` / `FREELANCEBASE_PASSWORD` が設定されていること
- Playwright がインストールされていること
- 共通ログインモジュール `~/.claude/snippets/playwright_freelancebase.py` が存在すること
- 企業検索・コメント・CRUD safety は `~/.claude/snippets/freelancebase/` を優先すること
- 画面仕様は `~/.claude/docs/freelancebase/` を参照し、書き込み系の追加時は `OperationPreview` と明示承認を挟むこと

**実行**

スクリプト: `.claude/skills/sales-new-client-meeting-prep/scripts/fb_register_company.py`

```bash
python3 .claude/skills/sales-new-client-meeting-prep/scripts/fb_register_company.py \
  --company-name "{企業名}" \
  --homepage "{コーポレートサイトURL}" \
  --phone "{代表電話番号}" \
  --contact-name "{担当者氏名}" \
  --contact-email "{担当者メール}" \
  --report-path "{タスクフォルダの絶対パス}/report.md" \
  --headless
```

**オプション**
- `--company-type`: `エンド` / `元請け` / `パートナー` / `その他`（指定しない場合は未選択）
- `--contact-phone`: 担当者の電話番号
- `--self-owner`: 自社担当者1 にセットする氏名（デフォルト: `赤木 宏志`）
- `--dry-run`: 書き込み前に流れを確認したい場合
- `--headless`: ブラウザを画面表示せず実行（推奨）

**挙動**
1. クイック検索で社名検索
2. ヒットあり → 既存企業を使用 / ヒットなし → 新規作成（企業名・HP・電話・自社担当者1）
3. 「企業担当者」タブ：同名 OR 同メールアドレスの担当者が既にいる場合は標準出力で重複を表示して y/n を確認
4. 「コメント」タブ：`report.md` 全文を投稿
5. 詳細画面 URL を標準出力に出力

**完了報告**

スクリプトが出力した企業詳細 URL をユーザーに伝える。

### ステップ8: report.md を Git にプッシュ

FreelanceBase へ登録した後も `report.md` は削除せず、タスクフォルダに残したまま Git にプッシュして保管する。

work-repo のブランチ運用（GitHub Flow）に従い、以下をセットで実行する:

1. feature ブランチ（既存タスクの worktree／ブランチがあればそれを使用、なければ `chore/<task-id>`）に `report.md` を含むタスクフォルダをコミット
2. push
3. `gh pr create` で PR 作成 → `gh pr merge` で main へマージ

`main` への直接 commit / push は禁止。必ず PR 経由で反映する。
