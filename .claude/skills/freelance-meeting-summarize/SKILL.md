## 概要

登録面談の Google Meet AI議事録（Docs）／録画（mp4）／面談メモ／スキルシート（PDF/Excel/Word）を 1 候補者単位で統合し、FreelanceBase に貼り付けられる形式の Markdown サマリを `output/` に保存する。

出力フォーマットは [SUMMARY_TEMPLATE.md](SUMMARY_TEMPLATE.md) を厳密に遵守する。

## ワークフロー進捗

このチェックリストをコピーして進捗を追跡する:

```
- [ ] ステップ1: 入力ソースの受け取り
- [ ] ステップ2: 各ソースのテキスト化
- [ ] ステップ2.5: FB 既存プロフィール突合（差分はユーザー確認）
- [ ] ステップ3: サマリ生成（提案単価はインボイス状況で+10/+15万）
- [ ] ステップ4: セルフチェック
- [ ] ステップ5: 出力・報告
- [ ] ステップ6: CRM書き込み（オプション）
```

## 前提

- カレントディレクトリはタスクフォルダ配下（`output/` を書き込む）
- 実行環境: Claude Code 上の手動実行。自動スケジューリングはしない
- 判断ロジックは Claude 本体が担う。Python スクリプトはテキスト抽出のみ

### 依存

```bash
pip3 install pdfplumber python-docx openpyxl openai google-api-python-client google-auth
brew install ffmpeg  # 録画を渡す場合のみ必要
```

- Google Docs API 認証: `~/.config/claude-gdocs-token.json`（既設、`~/.claude/snippets/google_workspace.py` が参照）
- Whisper: `OPENAI_API_KEY` 環境変数（録画を渡す場合のみ必要）
- FreelanceBase 共通基盤: `~/.claude/snippets/playwright_freelancebase.py` と `~/.claude/snippets/freelancebase/`
  - 候補者検索・コメント取得・API 捕捉・CRUD safety は共通 helper を優先する
  - 画面仕様は `~/.claude/docs/freelancebase/` を参照し、SKILL.md に重複記載しない

## 入力ソース

| # | ソース | 渡し方 | 必須/任意 |
|---|--------|--------|---------|
| 0 | 管理用ID（MJC...） | 文字列 or タスクフォルダ名から抽出 | **必須**（自動取得の検索キー） |
| 1 | Google Meet AI 議事録 | **既定: Drive から管理用IDで自動取得**（明示指定も Google Docs URL で可） | 任意（欠落時は「不明」） |
| 2 | 面談の画面録画 mp4 | **既定: Drive から管理用IDで自動取得**（明示指定もファイルパスで可） | 任意 |
| 3 | 面談メモ | **既定: CRM コメントタブの最新1件を自動取得**（明示指定もテキスト/パスで可） | 任意 |
| 4 | スキルシート | **既定: Drive から管理用IDで自動取得**（明示指定もファイルパスで可） | 任意 |
| 5 | FreelanceBase 既存プロフィール | **既定: 管理用IDで index API から自動取得**（`fetch_profile.py`） | 任意（欠落時は「新規登録」として扱う） |

欠落ソースがあってもスキルは動作する（当該部分は「不明」で埋める）。

**FB 既存プロフィール（ソース5）は必ず取得してからサマリを作る。** すでに登録されている情報は確度が高いが、面談で取れた最新情報の方が新しい。両者に**差分がある場合はどちらを優先するかを必ずユーザーに確認**する（勝手にどちらかへ倒さない。詳細はステップ2・ステップ3を参照）。

**前提運用**: Google カレンダーの登録面談イベントの題名に管理用ID（MJC...）を事前に入れておく。Gemini が議事録 Doc と Meet 録画 mp4 をこの題名ベースのファイル名で自動保存するため、管理用IDで一意に引ける。

## 手順

### ステップ1: 入力ソースの受け取り

ユーザーから以下を受け取る（引数で渡されていない場合は対話的に確認）:

- Google Docs URL（必須）
- 録画パス（任意）
- 面談メモ（テキスト直貼り or パス）
- スキルシートパス（任意）

入力が揃ったら、使うソースと欠落ソースをユーザーに報告してからステップ2に進む。

### ステップ2: 各ソースのテキスト化

スキルのルートを `SKILL_ROOT` と表記する（`.claude/skills/freelance-meeting-summarize/`）。

**AI議事録（既定: Drive から管理用IDで自動取得）:**

ユーザーが明示的な Google Docs URL を渡していない場合:

```bash
DOC_URL=$(python3 $SKILL_ROOT/scripts/fetch_meeting_doc.py <管理用ID>)
python3 $SKILL_ROOT/scripts/fetch_doc.py "$DOC_URL"
```

- Shared Drive `1zSKyKBRB0lorObsu2P87Hu4EM81x-32k` 内で `name contains '<管理用ID>' AND mimeType = google-apps.document`
- 複数ヒットは `modifiedTime desc` で最新
- 未ヒット時は exit 2（議事録欠落として処理を続ける）
- 明示指定の URL がある場合は従来通り `fetch_doc.py "<URL>"` に渡す

認証失敗時はユーザーに `~/.config/claude-gdocs-token.json` の存在を確認し、未整備なら中断する。

**録画（既定: Drive から管理用IDで自動取得 / 任意）:**

ユーザーが明示的なローカル mp4 パスを渡していない場合:

```bash
REC_PATH=$(python3 $SKILL_ROOT/scripts/fetch_recording.py <管理用ID>)
```

- 同 Shared Drive フォルダ内で `mimeType = video/mp4` を検索
- 複数ヒット時は最新更新日を採用
- OS tmp に DL（数百 MB〜数 GB、時間がかかる）
- `OPENAI_API_KEY` が未設定なら文字起こし自体をスキップ（録画 DL もスキップして差し支えない）

取得した `$REC_PATH` を `transcribe.py` に渡す:

```bash
python3 $SKILL_ROOT/scripts/transcribe.py "$REC_PATH"
```

数分〜十数分かかる。完了を待つ。

**面談メモ（任意 / 既定は CRM コメント自動取得）:**

ユーザーが明示的に指定したテキストやファイルが無い場合、FreelanceBase の候補者詳細「コメント」タブの **最新1件** を面談メモとして使用する:

```bash
python3 $SKILL_ROOT/scripts/fetch_comment.py <管理用ID>
# または --internal-id <id> で直接指定
```

- 出力: 最新コメントの本文（body のみ）を stdout に1行または複数行で返す
- `--json` で `{author, timestamp, body}` の構造化取得も可能
- 0 件なら stderr に WARN、stdout は空（面談メモ欠落として処理を続ける）
- システム自動生成コメント（例「# 提案案件 詳細をみる」）が混ざる可能性があるため、**内容を確認して必要なら除外**する

明示指定が渡された場合:
- テキスト直貼りならそのまま使用
- ファイルパスなら Read ツールで読み込む

**スキルシート（任意 / 既定は Drive 自動取得）:**

ユーザーが明示的にファイルパスを渡さない場合、Drive の指定フォルダ内から管理用IDで検索・取得する。**サマリの `[スキルシート]` 行に貼り付ける URL（Drive `webViewLink`）もこのステップで必ず取得する**（`--with-url` 必須）。**管理用IDで未ヒットでも即「不明」にせず、`--name <氏名>` フォールバックを必ず付けて再取得を試みる**（運営が管理用ID無しのファイル名でスキルシートを置いているケースを救済する）:

```bash
RESULT=$(python3 $SKILL_ROOT/scripts/fetch_skillsheet.py <管理用ID> --with-url --name "<候補者氏名>")
SKILLSHEET_PATH=$(echo "$RESULT" | sed -n '1p')
SKILLSHEET_URL=$(echo "$RESULT"  | sed -n '2p')
echo "PATH=$SKILLSHEET_PATH"
echo "URL=$SKILLSHEET_URL"
```

- ハードコード: `FOLDER_ID=1xBLxRqRnnMVi8IYy1tbDQ9CD1BP4BftT`（Shared Drive）
- `name contains '<管理用ID>'` で検索、複数ヒットは `modifiedTime desc` で最新を採用
- **`--name <氏名>` フォールバック（重要）**: 管理用IDで0件のとき、氏名（フルネーム連結→姓のみ）でフォルダ内を再検索する。現物が見つかれば **ファイル名を `<管理用ID>_<元の名前>` にリネーム**（`[RENAME]` を stderr に出力）してから採用するため、次回以降は管理用IDで一意に引ける。氏名で複数ヒットは最新を採るが `[WARN]` を出すので取り違えに注意する
- stdout: `--with-url` 指定で 1行目=tmp パス, 2行目=`webViewLink`（Drive URL）
- Google Sheets/Docs/Slides は自動的に xlsx/docx/pptx にエクスポートしてから DL
- **既定で** 対象ファイルに「リンクを知っている全員 = 閲覧者」の共有権限を付与（`anyone` 権限が既に存在する場合はスキップ）。stderr に `[SHARE] anyone-reader: granted|already|error` を出力。付与を避けたい場合は `--no-share` を指定
- 管理用ID・氏名フォールバックの双方で未ヒット時のみ exit code 2、スキルシート欠落として処理を続ける（URL も「不明」）
- ユーザーが明示的にローカルパスを渡した場合は URL 取得不可 → サマリの URL 行は「不明」

取得したパスを `extract.py` に渡す:

```bash
python3 $SKILL_ROOT/scripts/extract.py "$SKILLSHEET_PATH"
```

`.pdf` / `.docx` / `.xlsx` / `.txt` / `.md` に対応。スキャン画像のみの PDF は抽出結果が空になるので、警告してユーザーに通知する（OCR は対象外）。

**取得した `$SKILLSHEET_URL` は必ず保持し、ステップ3でサマリの `[スキルシート]` 行に貼り付ける。** URL 欠落時のみ「不明」を書く。

### ステップ2.5: FB 既存プロフィール突合（必須）

サマリを作る前に、**FreelanceBase に登録済みのプロフィール情報を必ず取得して面談情報と突合**する。登録済みの値は確度が高い前提情報だが、面談で得た情報の方が新しいため、両者の突合が要る:

```bash
python3 $SKILL_ROOT/scripts/fetch_profile.py <管理用ID> --summary
# 内部ID が既知なら --internal-id <id> でも引ける
```

- candidates/index API のレコードを JSON で stdout に返す（`--summary` で主要フィールドを stderr に併記）。未ヒットは exit 2（FB 未登録として続行）
- 突合に効く主なフィールド（実 API のキー名）:
  - `birth_date` / `age`（**生年月日が入っていれば年齢は自動算出される**）、`gender_id`（0 = 未設定）、`nationality_type_id`
  - `work_styles_key_values`（稼働形態）、`business_day_key_values`（稼働日数）
  - `monthly_price_num`（希望単価）、`qualified_invoice_registration_number`（インボイス番号。null = 未登録）
  - `occupation_key_values`（経験職種）、`dev_process_ids`（担当工程）
  - `desired_location` / `desired_detail` / `desired_working_schedule` / `desired_other`（希望条件）
  - `station_key_values`（最寄り駅）、`traffic_source_key_values`（流入経路）
- **双方に値があって食い違う項目（経験職種・稼働形態・希望単価 等）は、勝手にどちらかへ倒さず「FB既存 / 面談最新 のどちらを優先しますか？」とユーザーに確認**してからサマリを確定する
- FB が空欄で面談情報がある項目は「補完」として扱い確認不要。FB にあって面談で触れられていない項目（生年月日 等）は FB を採用
- **希望条件は FB の希望欄（desired_location / desired_detail / desired_working_schedule / desired_other）と整合させる**（食い違ったまま出力しない）
- **年齢はサマリに書かない**（CRM が生年月日から自動算出する）

### ステップ3: サマリ生成

抽出した各ソース（AI議事録・録画・面談メモ・スキルシート・FB既存プロフィール）のテキストを統合し、[SUMMARY_TEMPLATE.md](SUMMARY_TEMPLATE.md) のフォーマットに従って Markdown を生成する。ステップ2で差分の優先確認が必要になった場合は、**確認が済んでから**サマリを確定する。

**提案単価の算出ルール（必ず埋める）:**

- 提案単価は **候補者の希望単価（税別）を基準に、インボイス登録状況で上乗せ額を変える**:
  - **インボイス登録済み** → 希望単価 ＋ **10万円/月**
  - **インボイス未登録** → 希望単価 ＋ **15万円/月**
- **税別**で表記する。候補者希望が税込の場合は `÷1.10` して切り上げで税別換算してから上乗せする
- インボイス登録状況は FB 既存プロフィールの `qualified_invoice_registration_number`（null = 未登録）と面談メモの「インボイス登録」項目で判定する
- 例: 希望80万円（税別）・インボイス未登録 → `80 + 15 = 95万円` → `提案単価：￥950,000（税別・140-180h）`
- **インボイス登録有無はあくまで提案単価を決める内部的な参考情報**であり、クライアントへの提案には関係しない。**[概要] や [特記事項] には書かない**（サマリ本文からインボイスの記載を除外する）

**重要ルール:**

- [概要] は「・」区切り、最大7項目・1行40字以内、**強みが伝わる表現**（単なる列挙禁止）
  - **同一語句（動詞・名詞）を隣接する項目で繰り返さない**（例: 「開発」「経験」を連続3行で使うのは NG。同義語や別観点で言い換える）
  - **本人の自己評価より強い表現に盛らない**。面談メモ・議事録に「得意ではない」「若干」「軽く触った程度」等の限定表現があるスキルを「設計」「主導」「推進」等の強い表現で書かない。スキルシートの自己アピール文と面談での実力評価が食い違う場合は**面談メモを優先**し、強い動詞は両方で裏付けが取れる場合のみ使う（過大表現は現場ミスマッチ・早期退場リスクに直結する）
- [経験スキル] は全スキルを網羅し、カテゴリごとに整理（例: `AWS → AWS（EC2/S3/Lambda/RDS）`）
  - **一次情報源はスキルシートの「■技術スキル」セクション**。プロジェクト単位の利用技術は補足に留める
  - **同一技術を複数カテゴリに重複掲載しない**。TypeScript / JavaScript など両属性を持つ言語は **フロントエンドに寄せる**。Aurora / RDS 等の AWS マネージドサービスは **DB ではなく インフラ（AWS 括弧内）** に記載する
  - 言語とフレームワークは明確に分離し、**「言語（FW1/FW2/...）」のスラッシュ区切り括弧表記**で列挙する（カンマでなくスラッシュ）。例: `Node.js（Nest.js）/ PHP（Laravel/Zend Framework）`、`JavaScript（React/Next.js/Vue.js/Nuxt.js/Angular）`
  - **並び順は「直近であるほど」かつ「使用年数が長いほど」高スコアとし、高スコアの技術を各カテゴリ行の前半（左側）に置く**。言語・フレームワーク・ミドルウェア（DB / インフラ等）すべてに適用する。スキルシートの案件を時系列で精査し、最新案件・長期利用の技術を左に寄せる
  - **候補者の主職種がシステムエンジニア／エンジニアの場合、コーダー・デザイナー寄りの周辺スキル（HTML / CSS / SASS / jQuery / Storybook / Photoshop / Illustrator 等）は出力しない**（ノイズになるため）。コーダー・デザイナー職の場合のみ記載する
  - **特定のツール名・製品名・ブランドを指さない汎用名称は載せない**（例: インフラ欄の `VPS`、`SSL`、`オンプレ` 等の総称はノイズになるため除外）。`Nginx` `Docker` `Terraform` `cron` のように具体的なツール/プロダクト名は記載してよい
- [特記事項] は病欠・短期退場・稼働の制約など**純粋な特記事項がある場合のみ**記載し、なければセクションごと削除
  - **希望条件に関わる事項（NG業種・NG案件・希望する案件の方向性・出社/リモートの条件・優先順位 等）は [特記事項] に書かず、【希望条件】の該当欄に記載する**。例: 「金融（ど真ん中）案件はNG」は特記事項ではなく [その他希望] に入れる
  - インボイス登録有無は [特記事項] に書かない（提案単価の内部参考のみ。上記「提案単価の算出ルール」参照）
- **【希望条件】の4ラベルは FreelanceBase のフィールドと 1:1・同一表記で一致させる**:
  - `■希望の作業場所` → `desired_location` / `wish_info.work_location`
  - `■希望の作業内容` → `desired_detail` / `wish_info.work_content`
  - `■希望の作業時間` → `desired_working_schedule` / `wish_info.work_time`（**稼働時間帯の希望＝コアタイム/時短等**。週◯日の稼働日数ではない。時間帯希望が無ければこの行ごと削除）
  - `■その他希望` → `desired_other` / `wish_info.other_wish`（NG業種・NG案件・優先順位 等。ラベルは「その他**の**希望」ではなく「その他希望」）
  - **稼働日数（週●日）・稼働形態は希望条件テキストに入れない**。稼働日数は【基本情報】の ■稼働日数（FB の独立フィールド `business_day_ids`）、稼働形態は ■稼働形態 に書く
- **【希望条件】は FB 既存プロフィールの希望欄（`desired_location`/`desired_detail`/`desired_working_schedule`/`desired_other`）と整合を取る**。FB 側が空欄なら面談情報で埋め、双方に値があって食い違う場合はステップ2の差分確認に従う。サマリの希望条件と FB の希望欄が食い違ったまま出力しない
- **年齢は指定しない**。CRM は生年月日から年齢を自動算出するため、こちらから年齢を書く必要はない（[SUMMARY_TEMPLATE.md] に ■年齢 行は無い）。生年月日が読み取れた場合はステップ6で生年月日として書き込む
- [人材担当コメント] は 100 字以内で人柄・強み
  - **[概要] と表現が被らないように書く**（同じフレーズ「チームを牽引」等を繰り返さない）
- [経験職種] と [概要] は **相互整合** を取る（概要に「スクラムマスター」を書くなら経験職種にも含める、等）
- **[経験職種] に「リーダー」「チームリーダー」を含めない**。CRM occupation マスタの「エンジニアリングマネージャー」(value=32) と意味が違い、誤チェックの実害が出た。本物の EM（評価・採用・1on1 を担当）に限り「エンジニアリングマネージャー」と記載し、チームを引っ張った事実は [人材担当コメント] や [概要] で表現する（例: 「勉強会主導でチーム底上げに貢献」）
- 日本国籍の場合は【国籍】行ごと削除
- **`[スキルシート]` はラベルを単独行とし、`$SKILLSHEET_URL`（Drive webViewLink）を次行に改行して貼り付ける。** プレースホルダ「（URLを張り付ける）」のまま残さない。URL が取得できなかった場合のみ `[スキルシート]` の次行に `不明` と明記
- **`■スキル・経験サマリー` の直後に空行を 1 行入れてから `[スキルシート]` ラベル行を置き、その次行に URL を記載する**（見出しと URL を詰めない／ラベルと URL も同じ行に並べない）
- 読み取れない項目は **「不明」** と明記（空欄禁止）
- Canvas や添付ファイルは使わない（コードブロック内の直接テキストで出力）
- 「フォーマットに沿って提案資料を作成しました」等の前置きは禁止

出力末尾に以下の「使用ソース」メタ情報を付記する:

```

使用ソース:
- AI議事録: あり
- 録画: あり / なし（文字起こしをスキップ）
- 面談メモ: あり / なし
- スキルシート: あり（ファイル名） / なし
- FB既存プロフィール: あり / なし（新規登録）
```

### ステップ6: CRM書き込み（オプション）

v1 サマリ生成後、FreelanceBase 候補者詳細への自動書き込みを行う。**保存すると即時に社内表示に反映される**ため、差分プレビュー + 承認を必ず挟む。
新規・更新・削除系の変更を加える場合は `freelancebase.crud.OperationPreview` で対象・API・差分を表示し、既定 dry-run のまま確認できる形にする。

**6-1. 実行可否の確認**

- 「CRM（FreelanceBase）に書き込みますか？ (y/n)」とユーザーに確認
- `n` なら終了（v1 モード完了）

**6-2. 管理用IDの確定**

- v1 サマリの `■管理用ID：...` 行から `MJC...` 形式の ID を取得
- タスクフォルダ名に含まれていれば兼用可（例: `20260418-登録面談議事録作成_MJC90451111_TS/`）
- 不明 / 複数候補なら中断してユーザーに確認

**6-3. 中間 JSON への抽出**

- [SCHEMA.md](SCHEMA.md) の「抽出ルール（v1 サマリ → JSON）」に従って payload を構成
- プルダウン項目（稼働形態・経験職種・担当工程）は [masters.json](masters.json) の `aliases` を参照して value にマップ
- マスタに無いラベルは `meta.unknowns` に追加（書き込みスキップ）
- v1 サマリで「不明」の項目は JSON 上 `null` にし、CRM 書き込み時はスキップ（既存値を上書きしない）
- **営業開始日** (`sales_info.business_start_date`) は登録面談の対応日＝**今日の日付**を常にセット（年/月/日）
- **オンライン登録面談 録画URL** (`basic_info.recording_url`) は `fetch_meeting_doc.py` の戻り値（AI議事録 Doc URL）を既定でセット
- **連絡手段** (`basic_info.contact_method`) は面談メモの「連絡手段」項目から抽出（"LINE" / "メール" / "Slack" 等の自由記述）
- **氏名・フリガナ** (`basic_info.name_sei`/`name_mei`/`name_kana_sei`/`name_kana_mei`): 面談メモ・AI議事録から「岡村皓太（おかむら こうた）」のような表記を抽出。**書き込みは現在値が空のときだけ**（既存値は絶対上書きしない）
- **生年月日** (`basic_info.birth_date.year/month/day`): 面談メモの「生年月日」項目から抽出（例: `1996/10/9` → year=1996, month=10, day=9）。**書き込みは 3 つすべて未選択（"0"）の場合だけ**。生年月日が取れれば `age` は不要（CRM が自動換算）
- **年齢** (`basic_info.age`): **生年月日が取れる場合は基本的に省略**（`null` 推奨）。指定時も「現在値が空のときだけ」書き込む
- **提案方法** (`sales_info.proposal_method`) は面談メモの「先出し」項目から `先出しOK="1"` / `案件確認="2"` にマップ
- **稼働可能日数** (`sales_info.business_day_ids`): 面談メモの「週◯稼働」項目から構築（例: `週3~5日` → `["3","4","5"]`、`週5日のみ` → `["5"]`）
- **営業ステータス** (`management_info.sales_status`): 登録面談後は営業対象になるため、**既定で `"1"`（営業中）** をセット。意図的に維持したい場合のみ `null`
- **適格請求書発行事業者 登録番号** (`invoice_info.registration_number_13`): 面談メモ等にインボイス番号（`T` + 13桁）があれば、**`T` を除いた 13 桁数字のみ**をセット（例: `T3011401017348` → `"3011401017348"`）。記載が無ければ `null`。CRM では独立セクションで管理され、**空欄時のみ書き込む**（既存値は上書きしない）
- 生成した payload を `output/{candidate_id}_payload.json` に保存

**6-4. 差分プレビュー**

**6-5. 書き込み実行**

承認 (`y`) された場合のみ:

```bash
python3 $SKILL_ROOT/scripts/crm_write.py output/{candidate_id}_payload.json
```

- `--internal-id <id>` で ID 解決をスキップできる（管理用ID → 内部ID の変換にはクイック検索 → 詳細を見るの流れを自動実行）
- `--dry-run` で各ドロワーの開閉確認のみ（保存しない）
- `--notify {auto|on|off}` Slack 通知の制御（デフォルト `auto`: `$SLACK_WEBHOOK_NOTIFICATION_URL` があり dry-run でなければ通知）
- `--keep-screenshots` 失敗時スクリーンショットを tmp に残す（デフォルトは実行後削除。PII を含むため注意）
- セクション単位で成功 / 失敗を報告。1 つでも失敗があれば非 0 で終了

**6-6. 結果確認・通知**

- stdout の `=== 書き込み結果 ===` ブロックをユーザーに報告
- Slack 通知は自動で発火（`--notify auto` 既定）。本文は **管理用ID + セクションごとの成否のみ** で氏名・メール・電話は含めない
- 失敗セクションがあれば `tmp/crm_write_screenshots/` に screenshot が落ちる（PII を含みうるため **規定では実行後に削除**、`--keep-screenshots` で保持可）

**PII 取扱いルール**

- ログ出力: 管理用ID / 内部ID / セクション名 / API ステータスのみ。氏名・メール・電話・住所・最寄り駅の値は出さない
- screenshot: OS の tmp に退避、既定は自動削除。保持した場合も git には含めない（`tmp/` は .gitignore 対象）
- 編集ドロワーは `page.goto()` によるリロード禁忌（Vue SPA 状態が壊れる）
- 最寄り駅の autocomplete は v2 で対応済み。駅が取れている場合は対応する **都道府県** も自動で住所 select にセットする（`basic_info.prefecture_name`）
