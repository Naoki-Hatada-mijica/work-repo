## 概要

フリーランスボード（人材集客媒体・課金は応募ベース）の月次請求対象判定を半自動化する。
毎月月初に運営から前月分の応募者CSVが送られてくるので、それを入力として使う。
最終成果物は**返却CSV**（運営に返送する。C列=承認ステータス、D列=承認備考）。

`freelancehub-billing-check`（フリーランスHub版）の姉妹スキル。判定ロジックは共通だが、以下が異なる：

| 観点 | フリーランスHub | フリーランスボード（本スキル） |
|---|---|---|
| データ取得 | 管理画面をPlaywrightでスクレイピング | **ユーザーが渡すCSVを読む** |
| 管理画面操作 | ステータス一括変更あり | **不要**（管理画面ログイン自体が無い） |
| 年齢基準 | 50歳以上 | **60歳以上** |
| 運営への報告 | 管理画面ステータス変更で完結 | **返却CSVを生成、ユーザーが手動でメール返送** |
| FBメモ参照 | 未実装 | **実装済み**（コメントタブを取得して判定材料にする） |

### 入力
- **応募者CSVのパス** — 必須。ユーザーが毎回指定する
  - 例: `~/Downloads/FreelanceBoard_応募者一覧_20260501~20260531_株式会社mijica.csv`
  - 対象期間はファイル名（`YYYYMMDD~YYYYMMDD`）から自動抽出
  - **CSVは実行前にタスクフォルダの `input/` にコピーして保全する**

### 出力
- `02_task/{今日のタスクフォルダ}/output/{YYYY-MM}_フリーランスボード請求チェック.md`
  - 3セクション（請求対象外候補 / 要判断 / 請求対象）+ 返却CSVプレビュー
- 同名 `.html`（クリック可能リンク・折りたたみSlack/FBメモ履歴・色分けバッジ・**返却CSVプレビュー表**）
- `{YYYY-MM}_freelanceboard_candidates_{timestamp}.json`（実行結果）
- **返却CSV** `{元のCSVファイル名}_返却.csv`（ユーザーレビュー後に Phase 6 で生成）

## 判定の基本方針（最重要）

**CSVの情報だけで判断しない。** Slack #inquiries の過去メッセージと FreelanceBase の
メモ（コメント欄）にある過去情報が一番重要な判断材料。全候補者について必ず両方を参照する。

- CSV = 候補者の自己申告プロフィール（一次情報）
- Slack / FBメモ = 弊社側の過去の対応履歴・評価（**判定の決定打**）
- 突合に失敗・未実施の候補者は要判断に分類され、レポートに ⚠ 付きで明記される
- 自動判定は一次振り分け。**最終決定はユーザーがレポート（特にSlackスレッド・FBメモ全文）を見て行う**

## 前提条件

### 環境変数（`~/.zshrc` で永続化済み）
- `SLACK_USER_TOKEN`（Slack #inquiries 検索用、`xoxp-`）— **必須。未設定時はエラー終了**
- `FREELANCEBASE_EMAIL` / `FREELANCEBASE_PASSWORD`（FB突合用）

### 関連サービス
| サービス | 用途 |
|---|---|
| フリーランスボード | 応募者CSVの提供元・返却CSVの返送先（管理画面アクセスは無い） |
| FreelanceBase (https://freelancebase.jp/) | 営業ステータス・メモ突合 |
| Slack #inquiries | 応募者名で問い合わせ履歴を全文検索 |

## 実行フロー

```
Phase 1〜5（自動）: CSV読込 → Slack突合 → FB突合 → 判定 → レポート出力
   ↓
ユーザーレビュー: HTMLレポートを確認、要判断の候補者の承認/非承認を決定
   ↓
Phase 6（自動）: 返却CSV生成（--make-csv）
   ↓
ユーザーが運営にメール返送（手動）
```

### Phase 1: CSV読込
- UTF-8（BOM付き）・27列のCSVをパース
- 同一人物（氏名+メール一致）の複数応募は1候補者にまとめる
- 生年月日から応募日時点の年齢を計算

### Phase 2: Slack #inquiries 検索（必須）
- `search.messages` API を `SLACK_USER_TOKEN`（xoxp）で直叩き
- **3段クエリOR検索**（表記揺れ対応）: `"姓 名"` / `"姓名"` / `"姓"`+本文フィルタ
- ヒットごとに `conversations.replies` でスレッド全文取得（リプライが実質的な判断根拠）
- レート制限: 呼び出し間隔 3.2s + 429時 Retry-After リトライ（Tier2 = 20回/分）

### Phase 3: FreelanceBase 突合
- **3a. ステータス確認**: `/api/enterprise/candidates/index` に `keyword=氏名` で検索、完全一致候補の `sales_status_id` を確認
  - **`{1, 2}`（営業中/営業終了）= 課金対象**: 営業活動を実施済みのため、他の判定理由に関わらず**請求対象（承認）**とする
  - **`{3, 4}`（営業不可/取引停止）= 非承認の判定材料**: 過去流入の証跡と組み合わせて判定する
- **3b. メモ取得**: 氏名一致した全候補者（ステータス問わず）の詳細ページ
  （`/enterprise/candidates/{id_by_enterprise_id}`）でコメントタブ（`a[href='#comment-tab']`）を開き、
  `comment_candidates` API レスポンス捕捉 + DOM スクレイピングでメモ全件を取得（重複排除あり）

### Phase 4: 判定

**最優先ルール: FB 営業中/営業終了 → 請求対象（承認）**
営業活動を実施済みの人材は課金が正当なため、下記の判定理由に該当しても請求対象とする
（レポートには「※他の判定理由該当あり」として記録は残す）。

| # | 内部理由 | 判定ロジック | 分類 | 承認備考（D列）の表記 |
|---|---|---|---|---|
| 1 | 営業不可（FB） | FB営業不可/取引停止 + 過去流入の証跡（Slack 14日以上前 or 応募日以前のFBメモ） | 請求対象外候補 | 登録済営業不可人材 |
| 2 | 60歳以上 | 生年月日から応募日時点で60歳以上 | 請求対象外候補 | 60歳オーバーのため |
| 3 | 正社員（副業推定） | 現在の状況=正社員 + 週1〜2日、または Slack/FBメモで副業言及 | 請求対象外候補 | 正社員副業のため |
| 4 | 海外在住 | 地域に海外地名、または「その他」+ 海外シグナル（言語/自己PR/Slack/FBメモ） | 請求対象外候補 | 海外在住のため |
| 5 | 稼働日数不足 | 稼働日数が週1〜2日のみ | 請求対象外候補 | 稼働日数不足のため |
| 6 | 経歴詐称 | 自己PRの不自然な記述、または Slack/FBメモで言及 | 請求対象外候補 | 経歴に虚偽の疑いがあるため |
| 7 | 日本語能力不足 | 言語欄に日本語なし / 自己PRが英語のみ / Slack・FBメモで言及 | 請求対象外候補 | 日本語能力不足のため |
| 8 | 過去流入 | Slack に応募日14日以上前の履歴、または応募日14日以上前のFBメモ | 請求対象外候補 | 過去に別経路でご紹介済みのため |
| 9 | 経験不足 | Slack / FBメモで経験浅さに言及 | 請求対象外候補 | 実務経験不足のため |
| 10 | ヒューマンNG | Slack / FBメモで強NG言及 | 請求対象外候補 | 対応不可人材のため |
| 11 | 営業行為 | 自己PRが宣伝・勧誘 | 請求対象外候補 | 営業行為のため |

要判断送り（CSV生成前にユーザーのレビューが必須）:
- 正社員 単独（フリーランス転向の可能性）
- 住まいの地域=「その他」単独（海外シグナルなし）
- FB営業不可/取引停止 単独（過去流入の証跡なし＝今回応募の書類落ち結果の可能性）
- Slack / FB 突合の失敗・未実施

### 判定上の注意（Hub版から継承）
- **「お見送り」単独では請求対象外にしない**（見送り理由が上記基準に該当する場合のみ）
- 「見送り / お見送り / 対応なし」はヒューマンNGキーワードに**含めない**
- CSVの「応募種類」（通常案件/エージェント）・「経験スキル」欄は判定に**使わない**（自己申告のため）

### Phase 5: レポート出力
- HTMLレポート先頭に**返却CSVプレビュー**（氏名 / 承認ステータス / 承認備考）を表示
- 要判断は「⚠ 要レビュー」と表示され、解決されるまで返却CSVは生成できない
- 要判断カードには「非承認と判断する場合の承認備考候補」を併記

### Phase 6: 返却CSV生成（ユーザーレビュー後）
- 要判断の候補者について、ユーザーから承認/非承認（+理由）の指示を受ける
- 指示を overrides JSON にまとめて `--make-csv` を実行
- 返却CSVの仕様:
  - 元のCSVと同一の行構成・列構成・エンコーディング（UTF-8 BOM付き）
  - C列「承認ステータス」: `承認` / `非承認`
  - D列「承認備考」: 非承認理由（承認時は空欄）
  - 同一人物の複数応募行には同じ値を記載
- **要判断が未解決のままだとエラーになる**（レビュー必須の強制）

## 実行コマンド例

```bash
# Phase 1〜5: 分析・レポート出力
/usr/bin/python3 ~/.claude/skills/freelanceboard-billing-check/scripts/run.py \
  --csv "$HOME/Downloads/FreelanceBoard_応募者一覧_20260501~20260531_株式会社mijica.csv" \
  --output-dir /path/to/task/output \
  > /tmp/freelanceboard_check.log 2>&1 &

# Phase 6: 返却CSV生成（ユーザーレビュー後）
# overrides.json 例: {"山田 太郎": {"status": "非承認", "reason": "60歳オーバーのため"}, "佐藤 花子": {"status": "承認"}}
/usr/bin/python3 .../scripts/run.py --make-csv \
  --report /path/to/output/2026-05_freelanceboard_candidates_YYYYMMDD_HHMMSS.json \
  --overrides /path/to/output/overrides.json \
  --output-dir /path/to/task/output

# FB突合のみスキップ（テスト用。本番運用では使わない — FBメモは必須の判定材料）
... --skip-fb

# 期間をファイル名から抽出できない場合
... --month 2026-05
```

**注意**: pyenv の python3.12 には playwright が無いため、必ず `/usr/bin/python3`（system Python 3.9）を使う。

## 実行後の検証（Claudeが行うこと）

1. ログ（`/tmp/freelanceboard_check.log`）で全候補者に Slack / FB 突合（ステータス＋メモ）が
   実行されたことを確認する（`[N/総数] 氏名: Slack X 件` / `FB一致 X 件` / `FBメモ X 件` の行が全員分あるか）
2. レポートに「⚠ 突合失敗」「⚠ 突合が未実施」がある場合、原因を確認して再実行を提案する
3. 判定内訳（請求対象外候補 / 要判断 / 請求対象）をターミナルに要約して報告する
4. HTMLレポートをユーザーに送付し、要判断の候補者のレビューを依頼する
5. Phase 6 実行後、返却CSVの行数が元のCSVと一致していることを確認する

## 実行後のユーザー対応フロー

1. HTMLレポートを開いて判定結果をレビュー（特に要判断の Slack スレッド・FBメモ全文）
2. 要判断の各候補者について承認/非承認（非承認なら理由）を Claude に伝える
3. Claude が overrides JSON を作成して Phase 6 を実行 → 返却CSV生成
4. 返却CSVを確認し、フリーランスボード運営に**ユーザー自身がメール返送**
   （スキルからの自動送信は行わない）
