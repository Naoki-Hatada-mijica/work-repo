## 目的

人材をクライアントに提案する際、一部の案件では提案時に「必須スキル／歓迎スキルを満たしているか」を
要件ごとに○×で記入するよう求められる。本スキルは、案件要件と候補者サマリ（＋スキルシート）を突合し、
**各要件を ○／△／× ＋括弧コメント**で判定した表をコードブロックで出力する。

### 姉妹スキルとの使い分け

| スキル | 役割 |
|--------|------|
| **本スキル** `sales-candidate-skill-requirement-checklist` | **要件単位の ○／△／× 判定のみ**（クライアント提出の要件記入欄向け） |
| `sales-matching-score-diagnosis` | 適合度を100点満点で総合スコアリング・ランキング |
| `sales-talents-to-projects-matching` | 人材起点でオープン案件マスタ全件を逆引き |

総合スコアや複数候補比較ではなく、**1案件 × 1候補者の要件チェック表**が欲しいときは本スキル。

## 入力（ターミナル直貼り）

1回 **1案件 × 1候補者**。ユーザーがターミナルに以下を貼る（貼る順番は問わない。案件・人材どちらが先でもよい）。

- **案件側**: 【必須スキル】【歓迎スキル】の要件リスト。見出し表記は入力どおり尊重する。
- **候補者側**: 基本情報／営業情報／[概要]／[経験スキル]／[スキルシート] Drive URL 等のサマリ。

スキルシートの Drive URL があれば、その PDF/Excel/Word 本文も判定根拠に取り込む（次節）。

## ステップ1: スキルシート取得（Drive URL があれば）

候補者サマリに `[スキルシート]` の Google Drive URL があれば、本文を取得して判定根拠に加える。

```bash
# SKILL_ROOT は本 SKILL.md があるディレクトリの絶対パスに置き換える
# （例: <repo>/.claude/skills/sales-candidate-skill-requirement-checklist）
python3 "$SKILL_ROOT/scripts/fetch_drive_text.py" "<Drive URL または file ID>"
```

- **WebFetch / gws CLI は使わない**（認証不可。グローバル CLAUDE.md 準拠）。Drive 取得は必ずこのスクリプト。
- `.pdf` / `.docx` / `.xlsx` / `.txt` / `.md` と Google ネイティブ（Docs/Sheets/Slides）に対応。
- URL が無い／取得失敗（exit code 非0）／スキャン画像のみで本文が空（exit code 2）の場合は、
  **貼られたテキストのみで判定**し、後述の要確認点にその旨を明示する。
- 取得した本文・候補者情報は**外部（メール／Slack／PR本文／チーム共有物）に持ち出さない**。

## ステップ2: 要件の抽出

- 【必須スキル】【歓迎スキル】それぞれの要件を1行=1要件として抽出する。
- **非要件の説明行は除外する**。例:
  - 「▼以下は必須ではございませんが、ご経験の中にあるとよりマッチしやすいと考えております」
  - 「etc」「※」などの注記
- 要件文は**入力の文言・順序をそのまま使う**（要約・言い換えをしない）。

## ステップ3: 判定（○／△／×）

候補者サマリ＋スキルシート本文を根拠に、各要件を判定する。

### 記号の基準（3段階）

| 記号 | 基準 |
|------|------|
| **○** | スキルシート／サマリに**明確な根拠**がある。年数要件は記載年数が満たす場合。 |
| **△** | 部分的・個人学習レベル・隣接技術からの推測・年数がわずかに不足など、**条件付きで満たす**。 |
| **×** | 根拠が一切ない。 |

- **判定に迷う場合は △** とし、括弧で根拠の弱さを明示する（×で切り捨てない）。

### 年数・定量要件

- 「3年以上の◯◯経験」等は、サマリ／スキルシートの経験年数と突合する。
  - 明確に満たす → ○、わずかに不足／推定 → △、記載なし → ×。
- 経験年数が直接書かれない場合、職歴・案件期間から合理的に推定してよい（推定である旨をコメントに残す）。
- **PMO・マネジメント期間で実装ブランクがある場合はハンズオン年数を慎重に見る**。営業文の年数が
  スキルシートと食い違う場合はスキルシートを優先し、要確認点に挙げる。

### 特殊要件

- 「(外国籍の方) 日本語能力試験がN1レベル」等の**国籍前提条件**は、日本人なら `○（日本人です）` と注記。
- 複数技術を含む要件（例:「AWS・GCP などのクラウド」）は、いずれかを満たせば ○。片方のみ・浅い場合は文脈で ○/△ を判断。

### コメント方針

- **提案向け・前向きな口調**（クライアント提出を前提）。例:「DDDでの開発経験豊富です」「日本人です」。
- ただし**盛らない**。本人が「得意ではない」「若干」と述べているスキルを「設計」等の強い表現に格上げしない。
  △ の根拠は正直に書く（「個人で学習あり」「部分的な構築経験」等）。
- ○ で補足が不要なら**括弧コメントは省略してよい**（○単独行でよい）。

## ステップ4: 出力

- 【必須スキル】【歓迎スキル】の見出しでグルーピングする（入力にある見出しのみ）。
- 各行は `・{要件文} → {記号}（{コメント}）`。コメント不要な ○ は `・{要件文} → ○`。
- 順序・文言は入力の要件をそのまま使う。
- **出力はコードブロックで提示**する。

### 出力例

```
【必須スキル】
・3年以上のオブジェクト指向設計に基づいたサーバーサイドアプリケーションの開発および保守運用経験 → ○
・2年以上のJava/KotlinとSpringBootを使ったAPI開発業務経験 → ○
・5年以上のSQLを用いた開発業務経験 → ○
・自動テスト・自動デプロイを用いた開発業務経験 → ○
・Linuxの基礎的な知識や操作経験 → ○
・AWS・GCPなどのクラウドサービスを用いた3年以上の開発・運用経験 → ○
・コンテナサービスを使った開発・運用 → ○
・外部設計・詳細設計の作成業務経験 → ○
・同一サービスの1年以上の保守運用経験 → ○
・(外国籍の方) 日本語能力試験がN1レベルである → ○（日本人です）

【歓迎スキル】
・Go、PHPのプログラミング言語を用いた開発業務経験 → △（個人で学習あり）
・クリーンアーキテクチャ等の設計に関する理解 → ○（DDDでの開発経験豊富です）
・クラウド環境(AWS/GCP/Azure) の構築経験 → △（部分的な設定・構築、及び個人で学習あり）
・スクラムを用いた開発経験 → ○
```

## ステップ5: 要確認点の提示 → pbcopy

1. コードブロックの後に、**要確認点**を箇条書きで提示する。例:
   - 推定で判定した項目（年数を職歴から推定した等）
   - スキルシート URL が無い／取得失敗で**テキストのみで判定**した旨
   - 営業文とスキルシートで記載が食い違う項目
2. **最後に必ず「pbcopy しますか？」と確認**する。承認されたら、判定結果（コードブロックの中身＝
   `【必須スキル】`〜最終行までの本文。バッククォート ``` は含めない）を `pbcopy` に渡す。
   - コピー方法: 一時ファイル経由で `pbcopy < file`、または `cat <<'EOF' | pbcopy ... EOF`。
   - 再判定のたびに最新の結果で再確認する。
