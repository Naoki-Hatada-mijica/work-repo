## 概要

X運用代行会社が作成した投稿下書き（Google Spreadsheet）をレビューし、
フィードバックコメントをシートのI列に書き込む。

## ステップ1: シートデータ取得

1. 認証・データ取得は `~/.claude/snippets/google_workspace.py` の `get_sheet_values()` を使う（OAuth2トークンファイル `~/.config/claude-gdocs-token.json` 経由、非対話で動作）:

```bash
python3 -c "
import sys, json
sys.path.insert(0, __import__("os").path.expanduser("~/.claude"))
from snippets.google_workspace import get_sheet_values
rows = get_sheet_values('1yP5I9sTl-1BXf0Gb5YlfYNKTacZNI6gMQgXXAskEyWU', '6月通常投稿!A1:L200')
print(json.dumps(rows, ensure_ascii=False))
"
```

2. トークンファイルが無い・期限切れの場合のみ、フォールバックで `gcloud auth login --enable-gdrive-access` の実行をユーザーに促す（対話入力が必要なためClaude側では完了不可）。

## ステップ2: 対象行の特定

シートの列構造:
- A列: 確認日メモ
- B列: No.
- C列: ツイート日
- D列: ツイート予約（TRUE = 投稿済み）
- E列: ツイート作成（下書き）
- F列: 文字数
- G列: 意図/狙い
- H列: カテゴリ
- I列: mijica様お戻し（フィードバック欄）
- J列: SocialBiz修正
- K列: 修正後
- L列以降: 参考情報

以下の条件すべてを満たす行を**レビュー対象**とする:
- D列が空またはFALSE（未投稿）
- I列が空（未フィードバック）
- E列に下書きテキストがある

また、**I列にコメントがある行**を「過去フィードバック参照用」として保持する。

## ステップ3: レビュー実行

### レビュー時の参照コンテキスト

レビューの前に、以下を把握しておく:

**赤木宏志のプロフィール:**
- 株式会社mijica 代表取締役（2022年4月創業）
- キャリア: ギークス株式会社で法人営業 → 株式会社RelicでWebエンジニア → フリーランスエンジニア → 創業
- 事業: ITフリーランスマッチング（エンジニアに強み）
- ターゲット: スタートアップ企業のCTO・PM・PdM・人事（SESはターゲット外）
- 強み: 元フリーランスエンジニアが運営/監修、設計力・構築力・コミット力
- 現在はエンジニアではない（元エンジニア）。社員も登録面談を行う

**最近注目されているテックトレンド（抽象論の投稿に比喩・類推として使う）:**
- **ハーネスエンジニアリング**: モデル本体ではなく、その周辺（プロンプト・ツール・権限・コンテキスト・フック・サブエージェント等）の設計でAIエージェントの成果が決まる、というAnthropicが提唱する概念。組織論・マネジメント論で「個人ではなく構造で成果を出す」比喩として強力。
- **コンテキストエンジニアリング**: LLMにどう文脈を渡すかで出力品質が大きく変わる、という考え方。情報伝達論・ビジョン浸透論との親和性が高い。
- **AIコーディングエージェント**: Claude Code / Cursor / Copilot 等の台頭により、エンジニアの役割が「コードを書く人」から「エージェントを使いこなす設計者」へシフト。採用基準の再定義論と接続しやすい。
- **サブエージェント・スキル・MCP**: 責務分割と標準化された外部連携で、1人のエンジニアが扱える範囲が広がっている。チーム編成・権限設計の比喩に使える。
- **モデル選定**: Opus（複雑推論）/ Sonnet（汎用）/ Haiku（高速軽量）の使い分け。人材配置論との類推に使える。

これらは投稿のテーマに合う時のみ、赤木が自然に語れる深さで織り込む。無理に全部を詰め込まず、1投稿に1トレンドを自然に繋げる程度に留める。技術名は明示する（元エンジニアの一次情報感を優先）。

**過去FBから学んだ赤木のスタンス:**
- 「うまい→上手い」など漢字表記を好む
- 相手を責めすぎない表現を好む
- 誰向けの発言かを明確にする
- 向き不向きの差異を明確に出す
- 言い切り型は好まない（断定しすぎず、柔らかい表現を使う）
- 非エンジニア的な表現は避ける（元エンジニアとしての視点を維持）
- AI活用は「丸投げ」ではなく「構成案を自分で固めてからAIに預ける」スタンス
- 事実と異なる引用・発言は厳禁

**赤木の思想・主張ポジション（内容の整合性チェック用。トーンではなく「何を正しいと考えているか」）:**

このセクションは「赤木が特定テーマについて持っている主張」を蓄積する。投稿の中心主張がこれと矛盾・浅い理解になっていないかを照合する。

- **AI-native とは何か**: 既存プロダクトの各機能にAIを足して効率化すること（＝AI-enhanced）ではない。AIが動くことを前提に土台（データモデル・ワークフロー・UIの前提）から設計し直し、**AIを抜いたら成立しない**状態を指す。両者を混同し、AI機能の追加を「AI-native」と呼ぶ投稿は、概念の取り違えとして必ず指摘する。
- （今後、赤木の主張が固まったテーマをここに追記していく）

**過去のフィードバック済み行を参照し、赤木のトーン・観点に合わせたレビューを行う。**

### チェック観点

各対象行について、E列（下書き）・G列（意図/狙い）・H列（カテゴリ）をもとにレビューする。

**OKの出し方に関する原則（最重要）**: 「必須チェックに引っかからない＝OK」ではない。特に「実践論」「思考」「スタートアップ論」等の主張・意見系の投稿は、欠陥が無いことではなく**中心主張に一段深い視点・赤木独自の角度があるか**を能動的に問う。表層的な通説・バズワードの受け売りのままなら、OKではなく変更案（問い直し）を返す。OKは「主張の中身まで踏み込んで見て、赤木の思想と整合し賛同できる」場合に限る。文章が綺麗で読みやすいだけでOKを出さない。

#### 必須チェック（不合格なら変更案を出す）

1. **日本語の自然さ**: 文法的に成り立っているか、読んで意味が通るか
2. **わかりやすさ**: よく読まないと伝わらない文章、回りくどい表現がないか
3. **意味不明な文章**: 意味が通らない文章があれば削除を提案
4. **スタンスの適切さ**: 過度に極端なスタンスをとっていないか
5. **配慮**: 特定の人やグループを傷つける表現がないか
6. **事実の正確性**: 赤木の経歴・会社情報と矛盾していないか
7. **主張の妥当性・概念理解の正確さ**: 投稿の中心的な主張が、通説の受け売り・バズワードの表層的な利用になっていないか。技術トレンドやAI概念を扱う場合、概念を取り違えていないか（例: 各所にAIを足すだけのものを「AI-native」と呼ぶ等）。また主張が上記「赤木の思想・主張ポジション」と矛盾していないか。**浅い・取り違え・矛盾があれば、文章が綺麗でも変更案で問い直す**。判断に迷う場合や赤木の主張が未登録のテーマでは、OKを出す前にユーザーに「この主張の方向で合っているか」を確認する
8. **偽装請負リスク**: フリーランスは業務委託（準委任）契約。雇用を連想させるワード（「勤怠」「雇用」「出勤」「労働時間」「就業」「解雇」等）が含まれていないか確認し、含まれていれば業務委託に適した表現に置き換える

#### 加点チェック（改善提案として出す）

9. **ターゲットインサイト**: CTO/EM/CEO/人事のインサイトに刺さる内容になっているか
10. **CTO・採用人事への訴求**: CTOやエンジニア採用を行っている人事が「この会社に相談してみたい」と思える内容になっているか
11. **AIトレンドとの接続（抽象論のみ）**: カテゴリが「実践論」「スタートアップ論」「思考」などの抽象テーマの場合、上記「最近注目されているテックトレンド」のうち文脈に合うものを比喩・類推として織り込めているか。「原体験」「個人エピソード」のカテゴリはこの対象外とし、赤木本人の経験の重みを優先する。1投稿に1トレンドを自然に繋げる程度に留め、技術名は明示する

### フィードバック形式

すべてのフィードバックの末尾に「by Claude Code」を付与する。

**問題なし:**
```
OK

by Claude Code
```

**問題あり:**
```
変更案）
--------
{修正後の投稿全文}
--------

{変更理由の説明}

by Claude Code
```

## ステップ4: ユーザー確認

全レビュー結果を一覧表示する。形式:

```
### No.XX（カテゴリ: YY）
下書き: {E列の冒頭30文字}...
フィードバック: {OK or 変更案}
```

ユーザー（赤木）に「この内容でシートに書き込んでよいですか？」と確認する。
修正指示があれば反映してから次のステップへ進む。

## ステップ5: シートへの書き込み

ユーザーの承認後、`google_workspace.py` の `_get_credentials()` を流用して Python で `batchUpdate` を呼ぶ:

```python
import sys
sys.path.insert(0, __import__("os").path.expanduser("~/.claude"))
from snippets.google_workspace import _get_credentials
from googleapiclient.discovery import build

service = build("sheets", "v4", credentials=_get_credentials())
service.spreadsheets().values().batchUpdate(
    spreadsheetId="1yP5I9sTl-1BXf0Gb5YlfYNKTacZNI6gMQgXXAskEyWU",
    body={
        "valueInputOption": "RAW",
        "data": [
            {"range": "6月通常投稿!I{行番号}", "values": [["{フィードバック内容}\n\nby Claude Code"]]},
        ],
    },
).execute()
```

- I列にフィードバックコメントを書き込む（末尾に「by Claude Code」を付与）
